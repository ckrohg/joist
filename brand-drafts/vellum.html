PROLOGUE · WHY THIS EXISTS

The AI website builders shipped the same bugs.

We read every public bug tracker, postmortem, and Trustpilot review of Wix AI, 10Web, Lovable, the half-dozen open-source Elementor MCP servers, Builder.ai. Same failure modes, repeating. We engineered Vellum around them by name.

For two years, the AI-website-builder category has converged on the wrong product: one-shot generation. Press a button, receive a finished site. The output looks impressive in demos and fails on contact with reality — edits regenerate the entire page, destroying prior work; widgets render broken because the agent emitted settings the renderer doesn't understand; cache layers serve stale HTML; humans editing in parallel get silently clobbered.

The msrbuilds Elementor MCP bug tracker is the canonical reading list. Issue #32: tool wrote justify_content when Elementor expected flex_justify_content; the tool returned success; the frontend rendered default alignment. The agent thought it worked. The user only found out later.

This isn't an edge case. It's the dominant failure mode of the category — and every prior attempt shipped with some variant of it. Vellum's architecture treats each of those failures as a hard product invariant.

"The first AI tool pitch I've read in a year that doesn't read like ChatGPT wrote it."
— A skeptical agency owner, red-team review

Vellum is what an AI builder looks like when the writers read the bug trackers first. Open source. Schema-validated. Audit-logged. Built so the agent's work is durable enough to hand to a client.

CHAPTER 01 · THE DISCIPLINE

Six bugs killed every prior attempt. We refuse each by name.

Each entry below is a failure mode documented in a public bug report. Each is matched to a Vellum invariant that makes it structurally impossible.

i.

No silent writes

The tool that returns {success: true} while the actual save never lands is the most damaging bug class in this category. Vellum requires read-after-write on every mutation: the verified post-save element is returned, never a success flag.

— msrbuilds/elementor-mcp #36
ii.

No schema drift

When the LLM emits a settings key the renderer doesn't recognize, the agent's work is invisibly wrong. Vellum validates every widget setting against the live introspected schema of that specific install before any write. Unknown keys return 422 with Levenshtein-1 and flex_*-aware suggestions.

— msrbuilds/elementor-mcp #32 (the canonical case)
iii.

No regenerate-clobber

"Redesign my homepage" should not delete three weeks of human work. Vellum permits surgical edits by element ID only; full-page regeneration as a single tool call is refused with 423 policy.plan_required.

— Wix Harmony, 10Web, documented user complaints
iv.

No round-trip clobber

Optimistic concurrency via canonicalized content hashes. Every read returns a hash; every write echoes the hash it was planned against. Human edits in the Elementor UI invalidate that hash — the agent's next write fails fast and re-reads. No mystery overwrites.

— A category-wide failure
v.

No cache drift

Elementor CSS regeneration, page-cache flushing (SG Optimizer, WP Rocket, LiteSpeed, WPE native), and CDN purging (Cloudflare) all deferred to scheduled events post-write, with a frontend verification fetch confirming the change before resolving.

— msrbuilds/elementor-mcp #30 §2.3
vi.

No credit-burn loops

Per-task token budget with cheap-mode fallback. After N retries on the same error class, escalate to human rather than spending another $4 of Anthropic credits proposing the seventh variant of a wrong fix.

— Lovable, Elementor AI, Hostinger Horizons
CHAPTER 02 · THE WRITE PATH

One spine. Five gates. Verify asynchronously.

DocumentWriter::save() is the only sanctioned write path. update_post_meta('_elementor_data', …) is grep-banned in the codebase. There is no second way to write — by design.

// What the agent sees on a malformed write — actual error envelope

POST /wp-json/vellum/v1/pages/123/patch
{ "expected_hash": "sha256:abc...",
  "ops": [{
    "op": "update_settings",
    "element_id": "6a637978",
    "settings": { "justify_content": "center" }
  }] }

→ 422 Unprocessable Entity
{
  "code": "schema.unknown_key",
  "message": "Widget 'heading' has no control named 'justify_content'.",
  "recovery_suggestions": [{
    "op": "update_settings",
    "args": { "settings": { "flex_justify_content": "center" } },
    "rationale": "Levenshtein-1 + flex_*-aware match"
  }]
}

// The MCP server surfaces this to Claude as conversation:
// "I tried justify_content but Elementor calls it flex_justify_content.
//  Should I use that instead?"

The synchronous portion of DocumentWriter::save() does the minimum: PolicyGuard refusals, schema validation, atomic revision snapshot, the actual save via Elementor's own Document::save(), and read-after-write verification. Everything expensive — CSS regeneration, page-cache flushing, CDN purging, frontend verification, webhook emission — defers to scheduled events so the REST handler returns in under 250ms even on a 5MB page.

The agent receives an immediate response with pending_verifications; a webhook fires when the deferred work completes. The Claude Code skill surfaces this as "Applied (verification pending)" then updates the user once the webhook arrives.

CHAPTER 03 · WHAT'S IN THE BINDER

An index of what Vellum actually does.

Eight chapters of capability in the v1.0 release. No "AI-powered" hand-waving — each section maps to specific endpoints in the spec.

SECTION i
Pages & Elements
Full CRUD with surgical patch operations. Container vs section legacy autodetect. Responsive completeness enforcement.
SECTION ii
Widget Catalog
Live introspection of every registered widget — including Pro and third-party (JetEngine, Crocoblock). Per-skin schemas.
SECTION iii
Brand Kit (Globals)
Persistent colors, typography, voice, audience. Auto-rewrite color literals to kit refs (delta-E < 5 matching).
SECTION iv
Theme Builder Templates
Headers, footers, single, archive, popups. Display conditions with priority. Pro features handled.
SECTION v
Media
Image gen routing (DALL-E / Imagen / Flux). SSRF defenses on URL-mode upload. WebP/AVIF conversion.
SECTION vi
Plan Mode
Multi-step plans require WP-admin approval with CSRF + approver-binding. Atomic execution. Rollback on any step failure.
SECTION vii
Audit Log
Hash-chained tamper detection. Client-facing changelog export (HTML/CSV/PDF). 90-day retention default.
SECTION viii
Fleet Operations
Bulk onboarding via connect --config sites.yaml. Broadcast brand-kit updates. Per-site operating modes.

A colophon, not a launch.

Vellum is in pre-release. v0.5 alpha in ~6 weeks. v1.0 OSS launch on wp.org in 14–16. We're publishing the specs in the open, taking the time, and refusing to ship until the discipline is real. Star the repo to be notified when the alpha is ready for a staging site.
