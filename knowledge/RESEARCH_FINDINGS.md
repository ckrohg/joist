# Joist Research Findings

Synthesis of four research streams (visual-fidelity metrics beyond SSIM; dynamic/animated/lazy capture; how leading AI cloners/builders work; Elementor internals + best practices) into an actionable plan for the Joist cloner.

**System under research — Joist:** a pipeline that clones any website into a round-trip-EDITABLE Elementor/WordPress page.
- **(1) `capture-layout.mjs`** — Playwright captures the live DOM into a typed box-tree (container/heading/text/button/image/svg/mockup/code/list/video/tabs/nav; each with measured x/y/w/h + typography + background).
- **(2) `build-absolute.mjs`** — emits ABSOLUTE-POSITIONED native Elementor widgets via the WP REST API (`Document::save`), real fonts via `custom_css`, images uploaded to WP media.
- **(3) `grade-sections.mjs`** — composite = `0.4*visual(SSIM+exact-pixel) + 0.3*editability(fraction of source text reproduced as selectable native text) + 0.3*structural-fidelity(source block-types reproduced as the correct Elementor widget type)`.

**State:** 9 corpus-gated recipes, honest mean ~0.82 across 7 sites, generalization confirmed (clones unseen sites at 0.76–0.85).

**The four walls we are stuck on:**
- **(A) VISUAL SATURATED ~0.82** — SSIM is too tolerant to see remaining per-element color/typography/spacing/position errors, so visual fixes do not move the score. We need a BETTER FIDELITY METRIC.
- **(B) DYNAMIC/ANIMATED/LAZY content** (Stripe hero animations, scroll-triggered carousels, lazy iframes, Framer client-render) is not captured by our headless single-pass approach.
- **(C) DESKTOP-PIXEL ONLY** — absolute positioning is not responsive.
- **(D) reverse-engineered Elementor kses/render/lazy-load rules by trial** — fragile, undocumented.

## Executive Summary — the highest-leverage unlocks tied to our walls

The four streams converge on a striking agreement: **the field-leading cloners (10Web, Builder.io, Locofy, Webflow, Framer) and the field-leading fidelity benchmark (Design2Code) independently arrived at the SAME two structures Joist is missing — a block-matched per-element fidelity metric, and a nested container tree.** Both are directly portable, and Joist has a structural advantage (it owns the typed capture-tree on both sides, so it needs no OCR).

**1. Replace the saturated SSIM visual term with a block-matched, per-concern fidelity metric (Wall A — highest leverage).** SSIM is *provably* blind to exactly Joist's residual errors: it is "almost insensitive to changes in brightness, contrast, hue and saturation" and saturates in high-similarity regimes ([MicroSSIM arXiv 2408.08747](https://arxiv.org/html/2408.08747v1)). The closest published analogue to Joist's grader — **Design2Code (Stanford SALT, NAACL 2025)** — explicitly abandoned single-image metrics for a block-matched per-element decomposition: CLIP gestalt + four low-level per-block scores (block-match area, text Sørensen-Dice, position center-distance, **CIEDE2000 color**), with a **symmetric, un-gameable area denominator** `matched / (matched + unmatched_pred + unmatched_ref)` ([Design2Code paper](https://arxiv.org/pdf/2403.03163); [repo](https://github.com/NoviScl/Design2Code); verified `metrics/visual_score.py`). Joist can match its own typed capture nodes (no OCR) and score per matched pair: **CIEDE2000 color** (perceptually anchored: full credit ≤ΔE2, zero ≥ΔE10), **typography** (family+size+weight+line-height — fixes the MEMORY note that the grader "ignores font family"), **position/size ratios**, **text-Dice**. Report each sub-score SEPARATELY (Applitools "match-levels" lesson: never one saturated scalar). This makes per-element color/font/spacing/position fixes finally MOVE the score.

**2. Structural container-inference pass between capture and build (Walls C + structural-fidelity — biggest architectural lever).** Every credible AI cloner emits a STRUCTURED NESTED tree, never flat absolute leaves. **Locofy's "Large Design Model"** ([arXiv 2507.16208](https://arxiv.org/abs/2507.16208)) decomposes Joist's exact problem into four specialist models: Design Optimizer (flat layers → nested auto-layout containers), Tagging/Feature Detection (node role → semantic tag), Auto Components (repeated structure → reusable component), Responsiveness Model (predict cross-screen behavior from one static design). **Builder.io Visual Copilot** infers flex/grid grouping "even if designs don't follow auto-layout conventions" — Joist's exact situation. **10Web (our closest competitor) hits our exact walls in mirror image:** it emits native Elementor widgets (editable) but drifts on "alignment, width, height, font size" and is silent on responsive — proving structure-vs-fidelity is unwon; the prize is BOTH. Infer rows/cols/grids from aligned bounding boxes and emit Elementor **flex/grid containers** (the documented responsive primitive) instead of N absolute leaves.

**3. Dynamic-capture upgrades: `animations:'disabled'` + WebGL `preserveDrawingBuffer:true` + two-phase DOM-quiet readiness (Wall B).** Playwright's `animations:'disabled'` is a purpose-built FREEZER (fast-forwards finite animations to completion, cancels infinite to initial state) — far stronger than the current `reducedMotion:'reduce'` ([Playwright #11912](https://github.com/microsoft/playwright/issues/11912)). Stripe-class WebGL heroes come back blank because the default `preserveDrawingBuffer:false` makes post-frame `readPixels` undefined — fix by hooking `getContext` to force `preserveDrawingBuffer:true` BEFORE `goto` ([webgl2fundamentals tips](https://webgl2fundamentals.org/webgl/lessons/webgl-tips.html)); this directly recovers the blank Stripe hero that `blankish()` currently discards. `networkidle` is the wrong readiness signal for SPA/Framer (never settles or fires early) — replace with two-phase `domcontentloaded` → MutationObserver DOM-quiet gate ([tsvbits recipe](https://tsvbits.com/wait-no-mutations/)).

**4. Render-diff self-correction loop (Walls A + B).** The documented practitioner pattern: render built page → `magick compare -metric AE` → scalar loss + a diff IMAGE highlighting divergent regions in red → vision model localizes ("badge is the wrong colour, avatar stack too low") → emits targeted setting deltas → rebuild → re-grade ([understandingdata pixel-diffs](https://understandingdata.com/posts/pixel-diffs-are-the-prototype-spec/)). This is what the joist-clone skill's grader loop should converge on; it pushes past one-shot ~0.82.

**5. Replace kses/render trial-and-error with the documented Elementor capability map (Wall D).** The exact rules are now confirmed primary-sourced: `_elementor_data` JSON meta + `_elementor_css` cache that goes stale and MUST be force-regenerated after direct REST writes; `_elementor_edit_mode=builder` is required or the frontend renders the `post_content` fallback (corroborates Joist's hard-won finding); kses runs `wp_kses_post()` on save for users WITHOUT `unfiltered_html` (admins on single-site bypass it; **multisite admins do NOT**); `<style>` is stripped (use the per-element Custom CSS field, compiled to `post-{id}.css`). Codify these as a typed constraint table + a `wp_kses_allowed_html` filter shipped in the Joist plugin, instead of re-discovering by eye.

**Honest scoping:** Wall A is fully addressable by measurement+keys. Wall B is *partially* addressed — determinism stabilizes the metric and WebGL/animation freezing recovers specific content, but full animated/client-render capture is a multi-pass capture-pipeline project. Wall C requires the container-inference architecture change (supervised, corpus-gated) — absolute-positioning is desktop-only by nature, not by bug. Wall D is now codifiable. **Grader rewrites and core build-output changes are SUPERVISED (corpus-gate, A/B); pure measurement additions and capture-only tweaks are AUTONOMOUS-SAFE.**

## Stream 1 — Visual-fidelity measurement beyond SSIM

**1. SSIM saturates and is structurally blind to exactly Joist's residual errors.** SSIM is "almost insensitive to changes in brightness, contrast, hue and saturation" and exhibits documented *saturation* in high-similarity regimes — precisely Wall A at ~0.82 ([MicroSSIM, arXiv 2408.08747](https://arxiv.org/html/2408.08747v1); [SSIM limitations survey](https://medium.com/@tsdhrm/ssim-measuring-what-we-actually-see-5350c4a0c025)). Two images with SSIM 0.73 vs 0.74 can have substantially different blur ([videoprocessing.ai metric-cheating](https://videoprocessing.ai/metrics/ways-of-cheating-on-popular-objective-metrics.html)). For a full-page screenshot a per-element color shift or 6px font change is a tiny fraction of the windowed luminance/contrast/structure product — the score is dominated by white background + gross layout, which Joist already gets right. **This is the root cause of Wall A: SSIM measures the wrong thing.**

**2. Design2Code (Stanford SALT, NAACL 2025) — closest published analogue — abandoned single-image metrics for block-matched per-element decomposition.** Five components: high-level **CLIP** embedding similarity + four low-level per-block scores — **block-match** (matched area ratio), **text** (char-level Sørensen-Dice), **position** (block-center distance), **color** (**CIEDE2000**) ([paper](https://arxiv.org/pdf/2403.03163); [project](https://salt-nlp.github.io/Design2Code/); [repo](https://github.com/NoviScl/Design2Code)). Verified algorithm from `metrics/visual_score.py`:
- Detect text blocks in BOTH screenshots OCR-free (`get_blocks_ocr_free`), each = `{text, bbox normalized, color mean-RGB}`.
- Cost matrix from text similarity (`SequenceMatcher.ratio()`); solve with **Jonker-Volgenant / Hungarian** (`scipy.optimize.linear_sum_assignment`) + adjacency "consecutive bonus" to preserve reading order.
- **Filter matches with text similarity < 0.5** (low-confidence match treated as NO match).
- Per pair: `position_similarity = 1 - max(|Δcx|,|Δcy|)`; `color = max(0, 1 - ΔE2000/100)` via `colormath` (sRGB→Lab→`delta_e_cie2000`); text = Dice ratio.
- **Symmetric, un-gameable size score** (verified lines 494–558): `final_size_score = Σ matched_area / (Σ matched_area + unmatched_in_prediction + unmatched_in_reference)`. Denominator sums BOTH hallucinated AND missed blocks — can't win by over- or under-producing.
- Final = `0.2 * (size + text + position + color + clip)`; authors deliberately DO NOT collapse to one aggregate — each dimension is a separate diagnostic ([emergentmind](https://www.emergentmind.com/topics/design2code-benchmark)). **Joist's advantage: it built the Elementor tree and captured the source box-tree, so it needs NO OCR — match typed `capture-layout.mjs` nodes directly (deterministic, no OCR noise).**

**3. CIEDE2000 (ΔE2000) is the right color metric with hard perceptual anchors.** ΔE ≤ 1.0 imperceptible to most; 1–2 trained eye only; 2–3.5 commercial-acceptability limit; >5 clearly visible ([skychemi guide](https://skychemi.com/color-difference-formula-delta-e/); [Color difference, Wikipedia](https://en.wikipedia.org/wiki/Color_difference)). Lets Joist set a defensible non-arbitrary curve (full credit ≤ΔE2, zero ≥ΔE10) instead of SSIM's opaque product.

**4. The VRT industry converged on perceptual color distance + structure-aware tolerance, not raw pixels.** Playwright `toHaveScreenshot` uses **pixelmatch**, `threshold` default **0.2** is a *per-pixel YIQ* tolerance (not image-level ratio — widely misunderstood), image-level via `maxDiffPixels`/`maxDiffPixelRatio` ([Playwright SnapshotAssertions](https://playwright.dev/docs/api/class-snapshotassertions); [Wopee.io teardown](https://wopee.io/blog/screenshot-comparison-algorithms-visual-testing/)). pixelmatch + ODiff ship Vyšniauskas-2009 **anti-aliasing detection** (`includeAA:false`) so glyph edges don't create false diffs. Central thesis: **"highest-leverage intervention is environmental determinism, not algorithm choice"** — pin browser, disable animations, mock timestamps (relevant to Wall B).

**5. Applitools "Visual AI" is a *match-level* design, not one number — levels map onto Joist's dimensions.** Detects changes in "text, font, color, graphics, position" while ignoring platform pixel/AA noise; exposes **Strict / Layout / Content / Exact** ([match levels](https://applitools.com/tutorials/concepts/best-practices/match-levels); [UltimateQA](https://ultimateqa.com/applitools-match-levels/)). Takeaway: **report fidelity per concern (position/color/typography/text), never as one saturated scalar** — that is what makes a single-concern fix VISIBLE in the score.

**6. Learned metrics (LPIPS, DISTS) catch what SSIM misses but carry alignment + gameability risk.** LPIPS correlates better than SSIM/PSNR for semantic/texture diffs but "requires substantial compute," varies by pretrained model, and is misalignment-sensitive ([Patsnap](https://eureka.patsnap.com/article/perceptual-metrics-face-off-lpips-vs-ssim-vs-psnr); [explainer](https://www.tyrrell4innovation.ca/miword-of-the-day-is-learned-perceptual-image-patch-similarity-lpips/)). **DISTS** ([arXiv 2004.07728](https://arxiv.org/abs/2004.07728); [repo](https://github.com/dingkeyan93/DISTS)) is explicitly **tolerant to texture resampling + mild geometric misalignment** — ideal for an abs-positioned clone a few px off. **MS-SSIM** is the cheap no-dependency upgrade ("outperformed when evaluated by human observers", [emergentmind](https://www.emergentmind.com/topics/multiscale-structural-similarity-score-ms-ssim)). **FSIM** ([PolyU](https://www4.comp.polyu.edu.hk/~cslzhang/IQA/TIP_IQA_FSIM.pdf)) adds no decisive gain over MS-SSIM here.

**7. DOM/computed-style diff is the orthogonal render-independent half — and Joist already has the data.** DOM-based regression "compares DOM structure and computed styles… position, size, arrangement," immune to render noise, but "misses visual bugs that don't show up in the DOM" ([Bug0](https://bug0.com/knowledge-base/what-is-visual-regression-testing); [VRT best-practices](https://medium.com/@ss-tech/the-ui-visual-regression-testing-best-practices-playbook-dc27db61ebe0)). Robust answer is BOTH. Joist is uniquely positioned: `capture-layout.mjs` already records per-node x/y/w/h + typography + background, and can re-run on the clone — a free, perfectly-aligned box-tree on both sides.

**Caveats:** Do NOT copy Design2Code's OCR block-detection — use the typed capture-tree (keep the <0.5 text-match filter so a bad match never inflates color/position). Symmetry is non-negotiable (assert source-vs-source = 1.0 in the regression suite). Never let the loop optimize directly against a learned metric (LPIPS/DISTS/CLIP) — keep them as guardrails, not targets. ΔE on a single dominant color is lossy for gradients/images — restrict strict color to text + solid backgrounds, route images to DISTS/MS-SSIM. CIEDE2000 needs correct sRGB→Lab (D65, gamma-correct) — port a tested impl (`culori` or mirror `colormath`).

## Stream 2 — Capturing dynamic / animated / lazy / client-rendered sites

*(Deltas against what `capture-layout.mjs` / `capture-ensemble.mjs` already ship: step-scroll, lazy-promote, `reducedMotion:'reduce'`, `content-visibility:visible` override, click-drive tabs/accordions, `document.fonts.ready`, dark-ancestor-bg walk, canvas/svg/mockup region-raster with `blankish` guard, N-pass best-of ensemble.)*

**1. Playwright's `animations:'disabled'` is a purpose-built FREEZER, not just a flag.** Stops CSS animations/transitions + Web Animations API: **finite animations fast-forwarded to completion (fires `transitionend`), infinite cancelled to initial state**, then resumed after capture — exactly the "freeze entrance animations to a representative end-state" Joist needs for Stripe-class entrances. Current `emulateMedia({reducedMotion:'reduce'})` is weaker — only affects animations that HONOR the media query; bespoke JS/scroll animations ignore it. ([jest-playwright #815](https://github.com/playwright-community/jest-playwright/issues/815); [Playwright #11912](https://github.com/microsoft/playwright/issues/11912); [houseful.blog](https://www.houseful.blog/posts/2023/fix-flaky-playwright-visual-regression-tests/))

**2. `networkidle` is the wrong primary readiness signal for SPA/Framer.** Modern apps keep sockets/beacons/polling/service-workers open so network-idle is "never reached or unpredictably long" — and fires EARLY on lazy/infinite-scroll before content exists. Converged pattern: **two-phase — lifecycle boot (`domcontentloaded`) → deterministic readiness wait (`waitForFunction`/selector/DOM-quiet) proving content present.** Current `goto(..., 'networkidle')` + 8s networkidle re-wait is exactly the fragile pattern warned against. ([WebCrawlerAPI](https://webcrawlerapi.com/glossary/playwright/how-to-fix-playwright-networkidle-misuse); [why-teams-avoid-networkidle](https://medium.com/@gunashekarr11/why-top-automation-teams-avoid-networkidle-and-what-they-use-instead-c0d1e9439dc4); [Firecrawl SPA](https://www.firecrawl.dev/glossary/web-scraping-apis/best-way-to-scrape-single-page-applications-spas); [BrowserStack](https://www.browserstack.com/guide/playwright-waitforloadstate))

**3. DOM-quiet (MutationObserver) is the most reliable "settled" signal for client-render + hydration.** Recipe: `MutationObserver(subtree+childList+attributes)`, bump counter per mutation, decrement on interval, resolve when zero for a quiet window — with a hard timeout. Beats fixed `waitForTimeout` (Framer/React hydrate at unpredictable times). Current pipeline uses fixed settles (1800/550/1200ms) + `|h-prevH|<30` height-stability; a real mutation-quiet gate is strictly more accurate AND lets the fixed waits SHORTEN. ([tsvbits full code](https://tsvbits.com/wait-no-mutations/); [Playwright #26618](https://github.com/microsoft/playwright/issues/26618))

**4. WebGL/canvas screenshots come back blank because of `preserveDrawingBuffer:false`** (the default). After a frame is presented, `readPixels`/`toDataURL` is undefined behavior and returns empty. Stripe-class animated gradient heroes are WebGL canvases → post-hoc region screenshot is blank → which is exactly why `blankish()` (>92% near-white) exists and DROPS them. Robust fix: **force `preserveDrawingBuffer:true` at context-creation** (hook `HTMLCanvasElement.getContext` in an init script BEFORE `goto`). ([webgl2fundamentals](https://webgl2fundamentals.org/webgl/lessons/webgl-tips.html); [Userback](https://support.userback.io/en/articles/5233475-collect-screenshots-on-webgl-or-3d-canvas))

**5. Effective-background color requires an ancestor walk + alpha compositing.** Browsers report `transparent` or `rgba(0,0,0,0)`; real visible color = walk up parents to first non-transparent bg, then alpha-blend translucent layers — naive walks break on abs-positioned elements (visual parent ≠ DOM parent). Current `darkAncestorBg()` does a bounded contains-walk but is narrowed to dark+solid+code/mockup only; general effective-color (light gradients, translucent tints) is unsolved → contributes per-element color error SSIM can't see (Wall A). ([SiteLint](https://www.sitelint.com/blog/how-to-get-the-actual-real-background-color-of-an-html-element); [DEV transparent vs rgba(0,0,0,0)](https://dev.to/pavel_polivka/determining-the-effective-background-color-17ld); [shoehornwithteeth algorithm](https://shoehornwithteeth.com/ramblings/2014/02/compute-a-dom-elements-effective-background-colour/))

**6. "Save rendered state, not source" tools all rely on JS-settle + lazy-promote then snapshot the live DOM** — Joist's existing architecture. SingleFile downloads deferred/lazy images + iframes; ArchiveBox runs **multiple capture methods per URL** (wget + Chrome DOM + SingleFile + screenshot + WARC) keeping ALL, treating none as authoritative. Transferable: **multi-method/multi-state ensemble + keep-the-best** — `capture-ensemble.mjs` does this for *text count* but not *visual completeness* or *scroll-state*. ([SingleFile FAQ](https://github.com/gildas-lormeau/SingleFile/blob/master/faq.md); [SingleFile](https://github.com/gildas-lormeau/SingleFile); [ArchiveBox](https://archivebox.io/); [GitHub](https://github.com/ArchiveBox/ArchiveBox))

**7. Scroll-position-dependent state is a capture hazard for abs-positioning.** Parallax/scroll-pinned/progress elements have DIFFERENT `getBoundingClientRect` at different scroll offsets. The walk runs at `scrollY≈0` after step-scroll, but elements transformed during scroll may report transient transforms. Auto-scroll recipes scroll to bottom (trigger lazy) then back to top, accepting resting-state geometry. ([Puppeteer auto-scroll](https://community.latenode.com/t/how-to-auto-scroll-in-puppeteer-until-all-dynamic-content-is-loaded/16807); [SitePoint IntersectionObserver](https://www.sitepoint.com/react-intersection-observer-lazy-load-infinite-scroll-animations/))

**8. Cheap interpretable per-element metrics for Wall A.** **EDOKS** (EMD + Oklab) separates geometric (shape/position) from chromatic (color) distortion transparently; **LASI** matches LPIPS at hand-crafted cost. Zero-ML version: per-element **ΔE (CIEDE2000/Oklab)** + per-element box IoU — exactly the errors SSIM averages away. ([EDOKS arXiv 2601.19680](https://arxiv.org/html/2601.19680v1); [SUSS arXiv 2512.03701](https://arxiv.org/pdf/2512.03701); [LASI arXiv 2310.05986](https://arxiv.org/pdf/2310.05986))

**Caveats:** `animations:'disabled'` re-freezes per screenshot (multiple passes bake the initial frame each time — fine, but an infinite-animation hero's "initial state" may look empty; combine with `preserveDrawingBuffer` + accept representative-not-live). `preserveDrawingBuffer:true` has small GPU cost + can change AA slightly (verify it doesn't perturb working captures; won't help `webgpu` contexts — [gpuweb #4356](https://github.com/gpuweb/gpuweb/discussions/4356)). MutationObserver-quiet never settles on perpetual-animation libs (typewriter/counters/marquee) — ALWAYS pair with hard timeout cap. `domcontentloaded`-primary can extract too early on slow-API SPAs — keep height-stability backstop. Generalizing the bg walk FLOODS dark/translucent themes (round-37 finding) — keep opacity≥0.9 stop + bounded hops + contains-check, corpus-gate. Per-element ΔE/IoU needs reliable matching — start report-only, validate on the section-raster 1:1 proof page. None of this fixes Wall C. Preserve per-source raster filename tagging (cross-site contamination fix) for any new raster paths.

## Stream 3 — How leading AI website cloners + builders actually work

**1. The single most important architectural difference: every credible tool emits a STRUCTURED, NESTED layout tree, never flat absolute leaves** — the central gap vs `build-absolute.mjs`.
- **Builder.io Visual Copilot:** three stages — (a) a model "trained with over 2 million data points [that] transforms flat design structures into code HIERARCHIES"; (b) open-source **Mitosis** compiler → framework-agnostic IR; (c) fine-tuned LLM adapts to target. Works "even if designs don't strictly follow Figma's auto-layout conventions" — i.e. it INFERS flex/grid from raw geometry, Joist's exact situation. ([figma-to-code-visual-copilot](https://www.builder.io/blog/figma-to-code-visual-copilot); [best-figma-to-code-plugin](https://www.builder.io/blog/best-figma-to-code-plugin))
- **Locofy "Large Design Model" (LDM)** — most valuable primary source ([arXiv 2507.16208](https://arxiv.org/abs/2507.16208)) — decomposes Joist's exact problem into four specialist models trained on >1M nodes: **(1) Design Optimizer** (flat layers → nested auto-layout containers), **(2) Tagging & Feature Detection** (classify node UI role + semantic HTML, "high precision"), **(3) Auto Components** (repeated structure → reusable component), **(4) Responsiveness Model** (predict cross-screen behavior from one static design). Reports "superior performance in node positioning, responsiveness and reproducibility" vs generic LLMs; metric = **"preview match score"**. ([thenewstack](https://thenewstack.io/locofy-launches-large-design-model-to-turn-designs-to-code/))
- **Webflow AI** builds on **Flowkit**: "Colors as global variables, typography through reusable text styles, layout spacing by consistent rules you update in one place," generates page+section STRUCTURE up front "rather than flat layouts" — design-system-first, opposite of pixel-pinned. ([ai-site-builder](https://webflow.com/blog/ai-site-builder))
- **Framer AI** auto-generates ALL breakpoints on creation with **breakpoint inheritance**: edits to the primary (desktop) layout "automatically cascade to other screen sizes"; flexbox/grid auto-layout, not absolute. ([framer.university](https://framer.university/blog/the-new-ai-workflow-for-building-websites); [designcode.io](https://designcode.io/framer-web-design-responsive-layout/))
- **Vercel v0** constrains output to a fixed high-quality vocabulary (shadcn/ui + Radix + Tailwind), "responsive by default," refined conversationally. Lesson: constraining the target component set raises BOTH fidelity and editability. ([mindstudio](https://www.mindstudio.ai/blog/what-is-vercel-v0); [datacamp](https://www.datacamp.com/tutorial/vercel-v0))

**2. 10Web (closest competitor) proves the structured path is necessary AND hits OUR EXACT WALLS in mirror image.** A hands-on teardown confirms it clones a URL into **native Elementor widgets** ("all text and images nicely converted to standard Elementor widgets") — but drifts precisely where Joist's absolute approach is strong: "alignment, width and height… aren't quite right," missing icons, broken button styling, lost borders, spacing issues; homepage + ≤4 pages only; menu links break; silent on responsive. The field-leading Elementor cloner trades pixel-fidelity FOR structure/editability and still only ~75%. **Neither side has won — the prize is structure + fidelity together.** ([bobthewebbuilder 10web teardown](https://bobthewebbuilder.com/10web-ai-builder-elementor-pro-alternative/))

**3. The field converged on ~75% one-shot fidelity** ("roughly 75%; developer refinement required"), and the biggest accuracy lever is INPUT STRUCTURE QUALITY — well-organized inputs convert far better than unstructured. Implication: invest in the capture-tree's structural cleanliness, not just measurement. ([sixtythirtyten](https://www.sixtythirtyten.co/blog/from-figma-to-code-ai-design-to-dev-workflows-in-2026))

**4. Design2Code benchmark = the battle-tested fidelity metric (solves Wall A).** Explicitly rejects SSIM-alone ("identical pixels can arise from different code; correct layouts can have pixel variation"); scores disaggregated axes: **Block-Match** (size-weighted recall/precision of matched text-block pairs by bbox overlap), **Low-Level Element Matching (LLEM)** = averaged recall across {existence, text, position, color} per matched block, + **CLIP-score** (ViT-B/32 cosine) + tree/HTML BLEU. Top commercial models only clear LLEM 80% — headroom where SSIM saturates. ([emergentmind](https://www.emergentmind.com/topics/design2code-benchmark))

**5. Dynamic/animated content (Wall B) handled two ways:** (a) **screen-recording-to-code** — abi/screenshot-to-code takes a VIDEO of a site in action → functional prototype (multi-frame beats single-pass); (b) **vision feedback loops** — render output, diff against target, read loss, self-correct. Vision models alone "struggle with precise location or color" in a single pass — why the loop matters. ([abi/screenshot-to-code](https://github.com/abi/screenshot-to-code); [SEJ GPT-4V limits](https://www.searchenginejournal.com/gpt-4-with-vision-examples-limitations-and-potential-risks/497250/))

**6. The render-diff self-correction loop is a documented practitioner pattern (attacks Walls A+B).** "Pixel diffs are the prototype spec": render at same viewport → `magick compare -metric AE` → scalar (e.g. normalized `0.665`) PLUS a diff image highlighting divergent regions in red → agent "opens it and sees… 'divergence concentrated in the right rail, avatar stack too far down, badge wrong colour'" → edits → re-screenshots → reads new loss. Self-supervises because there's "a scalar to minimize and a visual map of where to look." ([understandingdata](https://understandingdata.com/posts/pixel-diffs-are-the-prototype-spec/))

**Caveats:** Container inference is a precision/recall tradeoff — over-grouping invents nesting (breaks fidelity), under-grouping keeps flat leaves; Locofy's Design Optimizer model EXISTS because this is hard — start with conservative high-confidence clusters, keep absolute fallback, corpus-gate (4 prior non-improvements caught this way). CLIP/LLEM have blind spots (CLIP under-penalizes layout shifts) — combine with bbox-IoU, validate human-correlation on the 7-site corpus before trusting as optimization target. `magick compare -metric AE` is AA/subpixel sensitive — use `-fuzz`, per-section crops, prefer the diff IMAGE for localization over the raw scalar. Marketing pages lie about responsiveness — treat auto-responsive as a quality FLOOR. Code-emitters (v0/Builder/Lovable/bolt) target React/Tailwind not kses-filtered Elementor — their LAYOUT-INFERENCE transfers, their output substrate does NOT. Multi-frame capture has no clean "done" signal for infinite animations — define a steady-state heuristic. Locofy LDM exact algorithms are not public — treat the four-model decomposition as a DESIGN TEMPLATE, not a spec.

## Stream 4 — Elementor internals + best practices

**1. The `_elementor_data` model is exactly what Joist emits — meta keys + regeneration path confirmed.** Full page tree = JSON string in `_elementor_data` post-meta; generated per-page CSS in `_elementor_css` meta AND on disk at `/wp-content/uploads/elementor/css/post-{id}.css` ([criticalwp](https://criticalwp.com/blog/elementor-complete-guide/); [developers.elementor CSS rendering](https://developers.elementor.com/css-rendering-performance-improvement/)). Every node = `{id, elType, isInner, settings, elements}`; widgets add `widgetType`; `elType ∈ {container, section, column, widget}` ([general-elements](https://developers.elementor.com/docs/data-structure/general-elements/)). **Load-bearing:** `_elementor_css` "becomes stale after updates/migrations," rebuilt via Tools → **Regenerate Files & Data**. A tool writing `_elementor_data` directly via DB/REST MUST force CSS regeneration or the page renders with stale/empty CSS — corroborates Joist's `_elementor_edit_mode=builder` finding (frontend must render the tree, not the `post_content` fallback). ([regenerate-css-data](https://elementor.com/help/regenerate-css-data/))

**2. CSS is generated from per-widget scoped selectors keyed on element id.** Controls declare `selectors` arrays mapping `{{WRAPPER}} <css-sel> => prop: {{VALUE}}`; `{{WRAPPER}}` → `.elementor-element.elementor-element-<id>` (every instance scoped). Placeholders: `{{SIZE}}{{UNIT}}`, `{{TOP}}{{UNIT}} {{RIGHT}}…`, `{{URL}}`; responsive values emit media-query-wrapped rules ([selectors](https://developers.elementor.com/docs/editor-controls/selectors/)). **Implication: Joist's per-element fidelity is bounded by which control keys it sets — every invisible-to-SSIM delta (color/font/letter-spacing/padding) maps to a SPECIFIC settings key, so Wall A is partly addressable by widening the key set, not just the metric.**

**3. Containers (flexbox + grid) are the responsive primitive — the documented path off desktop-pixel-absolute (Wall C).** Containers replace section/column → "much less DOM output," per-breakpoint control of direction/order/alignment/distribution WITHOUT duplicating content ([flexbox-containers blog](https://elementor.com/blog/introducing-flexbox-containers/)). Confirmed responsive Layout controls + CSS mappings: Content Width (Boxed/Full), Width (px/%/vw per-viewport), Min Height (px/vh), Direction → `flex-direction`, Justify → `justify-content`, Align → `align-items`, Gap, Wrap ([size behavior](https://elementor.com/help/set-flexbox-container-size-behavior/); [layout-tab](https://elementor.com/help/container-layout-tab-settings/)). **Grid container** = 2D (template columns/rows fr/minmax/repeat, gap, auto-flow, span); use Grid for true multi-column regions — solves the multi-column overflow wall ([grid options](https://elementor.com/help/grid-container-layout-options/); [flexbox-vs-grid](https://wpbuilderaddons.com/tutorial/flexbox-and-grid-in-elementor/)). Section→Container conversion is ONE-WAY (duplicate-then-delete) — never round-trip a container page back to sections ([convert-sections](https://elementor.com/help/convert-existing-sections-to-containers/)).

**4. Breakpoints — exact defaults + cascade-down model Joist can exploit.** Defaults (post-3.2): **Mobile ≤767px (max-width)**, **Tablet ≤1024px (max-width)**, Desktop ≥1025; old `viewport_md`/`viewport_lg` → `viewport_mobile`/`viewport_tablet` ([3.2 changes](https://developers.elementor.com/breakpoint-system-changes-in-elementor-v3-2-0/); [browserstack](https://www.browserstack.com/guide/elementor-breakpoints)). Six optional custom breakpoints (3.4+): Mobile, Mobile Extra (~768–880), Tablet, Tablet Extra (~1025–1200), Laptop (~1366), Widescreen (~2400); all max-width EXCEPT Widescreen = min-width ([3.4 blog](https://elementor.com/blog/introducing-elementor-3-4/); [breakpoint gotchas](https://developers.elementor.com/additional-custom-breakpoints-technical-details-and-gotchas/)). **Cascade is one-directional (down):** unsuffixed = Desktop/base, applies to all smaller unless overridden; suffixed (`_tablet`/`_mobile`/`_widescreen`/`_laptop`/`_mobile_extra`/`_tablet_extra`) override DOWNWARD only ([seahawkmedia 2026](https://seahawkmedia.com/wordpress/elementor-breakpoints/)). Gotcha: with breakpoints experiment active, suffixed responsive values aren't readable via `get_settings_for_display()` in `render()` (doesn't affect REST authoring).

**5. Absolute positioning — confirmed model + confirmed responsive risk.** Advanced → Custom Positioning: Position Absolute/Fixed; Width Default/Inline/Custom; Horizontal (Left|Right)+offset; Vertical (Top|Bottom)+offset; units px/%/vw/vh/em per-breakpoint. **Absolute elements positioned relative to nearest positioned ancestor (parent container/section), NOT the page — "still relative to the column they're inside"** — the documented source of responsive breakage. Official guidance: use %/vw/vh (not px) + small values ([how-to-set-absolute-position](https://elementor.com/help/how-to-set-absolute-position-for-an-element/); [custom-positioning blog](https://elementor.com/blog/custom-positioning/); [academy units](https://elementor.com/academy/how-to-use-custom-positioning-units-for-best-mobile-responsive-behavior/)). **Confirms Wall C is inherent to pixel-absolute, not a bug.**

**6. kses / save / render — precise survival rules.** Users WITHOUT `unfiltered_html` (editor or below): ALL widget fields run `wp_kses_post()` on save → only the WP "post" allow-list survives (strips Heading `class`/custom attrs, image `srcset`, `<iframe>`/`<embed>`/`<script>`) ([#27939](https://github.com/elementor/elementor/issues/27939); [#16111 srcset](https://github.com/elementor/elementor/issues/16111); [#31651 embeds](https://github.com/elementor/elementor/issues/31651)). **Admins WITH `unfiltered_html` bypass kses — fields save raw. So Joist's REST writes survive intact IFF the auth user is an admin with `unfiltered_html` (true single-site; FALSE on multisite — even admins lack it).** This is the real rule behind "inline `style=` + nav/iframe/video/`role=` survive." To allow extra tags regardless of role, hook **`wp_kses_allowed_html`** (post context) ([gist iframe kses](https://gist.github.com/bjorn2404/8afe35383a29d2dd1135ae0a39dc018c)). `<style>` blocks stripped (`style` not a post-context tag) — supported channel is the per-element/page **Custom CSS** field (Pro), compiled into `post-{id}.css`, not injected as sanitizable `<style>` ([custom-css](https://elementor.com/help/custom-css-in-elementor/); [unfiltered_html](https://publishpress.com/blog/unfiltered-html-permission-wordpress/)).

**7. Global styles / V4 are now real CSS variables — useful for typography/color fidelity.** Global Colors + Fonts live in the **Kit** (`wp_options`), exportable as Site-Settings JSON; Elementor also reads `theme.json` palette/fontFamilies ([theme-style](https://elementor.com/help/theme-style-global-settings/); [global-style](https://developers.elementor.com/docs/editor-controls/global-style/)). **V4 / 3.33 (current):** Variables Manager → CSS variables (design tokens); **per-element Custom CSS now native** (Pro, element-isolated — no `<style>`/kses workaround); Blend Modes native; atomic CSS-first class-driven engine with `styles` array + `classes` (vs V3 inline `settings`) ([3.33 V4 blog](https://elementor.com/blog/elementor-333-v4-variables-manager-custom-css/)). Matches Joist memory: V3-on-V4 is the production shape; V4 auto-adds `styles`/`classes`/`interactions` on save.

**Caveats:** YouTube transcripts not fetchable — the Figma-to-Elementor "never use pixels" videos corroborate consensus (containers + %/vw/min-height + global styles) but no verbatim steps. Multisite breaks the kses assumption — PREFLIGHT `unfiltered_html`. Absolute is relative to PARENT not page — verify parent is a positioned ancestor (`position:relative`). CSS regeneration is MANDATORY after direct DB/REST writes. V3-vs-V4 divergence: per-element Custom CSS / Variables Manager / Blend Modes are V4/Pro-gated — detect V4+Pro first. ### Stream A — Visual/structural fidelity measurement, deeper (the ground-truth Design2Code port + 2025-26 benchmarks)

**1. Joist's perElement metric diverges from the published Design2Code `visual_score.py` in four scoring-meaningful ways** (read verbatim from [NoviScl/Design2Code `metrics/visual_score.py`](https://github.com/NoviScl/Design2Code/blob/main/Design2Code/metrics/visual_score.py)):
- **Color normalization is `/100`, not `/10`.** D2C: `similarity = max(0, 1 - delta_e/100)`. Joist's `colorScore` uses `1 - dE/10` with a `dE<=2 → 1` floor (`perelement-score.mjs:160-167`) — **~10× stricter**. A 20-ΔE off-brand color scores 0.0 in Joist vs 0.80 in D2C. Defensible (Joist *wants* human-strict) and the D2C divergence is the citation if anyone challenges the harshness — **do NOT relax to `/100`**; D2C's lenient color is a known weakness of that benchmark, not a target.
- **D2C MERGES blocks before matching** (`find_possible_merge` → `merge_blocks_wo_check`, `consecutive_bonus, window_size = 0.1, 1`). A source paragraph the clone renders as 2 wrapped lines (2 boxes) gets merged into one block to match 1:1. Joist has **no merge step** — its Hungarian matcher leaves the second fragment unmatched, dumping its area into `unmatchedCloneArea` and deflating `areaCoverage`. A real, un-modeled deflation source on text-heavy clones.
- **D2C uses a context-aware cost adjustment** (`adjust_cost_for_context`): a candidate match gets a bonus proportional to how well its spatial neighbors also match (`top_k` of the `window_size` window) — rewards reading-order / local-neighborhood preservation. Joist's matcher is purely pairwise (text-Dice + typo + center-dist), no neighborhood term.
- **D2C's CLIP term masks matched blocks via inpainting (`mask_bounding_boxes_with_inpainting` + `cv2.INPAINT_TELEA`) and CLIPs the RESIDUAL** — exactly the CSS-drawn illustrations + abs-positioned layers Joist's capture SKIPS (**wall D**). Joist has **no semantic/residual term at all**. Final D2C score is an unweighted mean `0.2·(size+text+position+color+clip)`.
- D2C position uses `1 - max(|Δcx|,|Δcy|)` on normalized 0-1 coords (`max_distance=1.42`). Joist's PASS-2 position normalizes Δcx/refW and Δcy/refH separately (`perelement-score.mjs:604-606`) and already has the **y-scale two-pass stretch tolerance D2C LACKS** — D2C wrongly penalizes a uniformly-taller correct clone; Joist already fixed this (ydrift matcher, kept).

**2. 2025-26 benchmarks add structural + interactive dimensions Joist's grader is missing:**
- **TreeBLEU** ([WebCode2M, arXiv:2404.06369](https://arxiv.org/html/2404.06369)): `TreeBLEU = |S(t) ∩ S(t̂)| / |S(t̂)|`, `S(·)` = set of unique 1-height subtrees (a node's tag + the tags of its direct children, attrs stripped). Pure DOM-hierarchy recall — catches **nesting** errors a flat block-type count can't (e.g. a card whose icon/title/body flattened into siblings). Computable from the built Elementor tree (container→widget parent/child tags). Recall-only + tag-only — treat as a *miss detector*, not precision; build the tag-equivalence map carefully.
- **DesignBench** ([arXiv:2506.06251](https://arxiv.org/html/2506.06251v3)): defect taxonomy maps 1:1 onto Joist's detector slots — 6 repair categories (*Occlusion, Crowding, Text-overlap, Alignment, Color-and-contrast, Overflow*) + 9 edit attributes weighted by real frequency (Change 55.8% / Add 38.1% / Delete 6.1%; component 27.9%, text 17.1%, position 16.0%, color 15.2%, shape 13.4%, size 10.4%). MLLM UI-issue-detection accuracy is only **0.27**; dominant failures are "wrong element size, element missing, wrong element position" — i.e. **geometry/coverage, not color**, is where the field fails. Validates tilting Joist's effort toward position/coverage/overflow detectors over more color tuning.
- **UI2Code^N** ([arXiv:2511.08195](https://arxiv.org/html/2511.08195v3)): reformulates UI-to-code as closed-loop "render → inspect → relative-rank → edit." Central result: **Relative Visual Policy Optimization (RVPO)** — pairwise tournament of N rendered candidates (`Wi := Σ o_ij` wins; failed renders = -1) — gives a more stable learning signal than any absolute scorer, beating Gemini-2.5-Pro and Claude-4-Sonnet on D2C/Web2Code; test-time scaling N=1→5 moved real-site fidelity 66%→74%. Actionable: **Joist's router/revise-loop should rank candidate builds by a RELATIVE A-vs-B render diff, not chase an absolute composite threshold** — the abs/flow router already picks "best-per-site," the right shape.

**3. ### Stream B — Dynamic/animated/client-render capture, deeper (THREE clocks, not one)

> Builds strictly beyond Stream 2 above (which already proposed `animations:'disabled'`, `preserveDrawingBuffer:true`, DOM-quiet MutationObserver gate, `domcontentloaded`-readiness, effective-bg walk).

**1. There are THREE animation clocks; the pipeline controls none deterministically — they need three different freezers.** (a) *CSS time-based animations/transitions* → frozen by Playwright `animations:'disabled'` (finite→end+`transitionend`, infinite→initial). (b) *JS-timer/rAF-driven* (typewriters, JS counters, GSAP `.to()` not on scroll, Lottie) → NOT touched by `animations:'disabled'`; need Playwright's **Clock API** (`page.clock.install()` mocks `Date/setTimeout/setInterval/requestAnimationFrame/requestIdleCallback/performance`; `clock.pauseAt()` + `clock.runFor(ms)`) ([Playwright Clock](https://playwright.dev/docs/clock)). (c) *WAAPI + CSS* → driven by `document.timeline.currentTime`, which the Clock API does NOT patch — open bug [Playwright #38951](https://github.com/microsoft/playwright/issues/38951): after a clock install, `document.timeline.currentTime` is unmocked so WAAPI animations actually BREAK. **Clock and `animations:'disabled'` are complementary, not redundant; Clock alone can corrupt WAAPI heroes.**

**2. `document.getAnimations()` (on `document`) is a stronger, more surgical freezer than `reducedMotion` OR `animations:'disabled'` — and unused.** Returns ALL CSS animations, CSS transitions, and WAAPI animations document-wide ([MDN](https://developer.mozilla.org/en-US/docs/Web/API/Document/getAnimations)). Each exposes `.finish()` / `.currentTime=ms` / `.pause()`. One line drives every entrance/reveal to its END (revealed) state — the representative final frame Joist wants — without depending on the site honoring `prefers-reduced-motion` (bespoke JS/GSAP/Framer reveals ignore `reducedMotion` entirely; this is why framer collapsed to ~4 leaves). `.finish()` throws on infinite-iteration animations → mandatory `try/catch → currentTime = activeDuration` fallback.

**3. Scroll-driven CSS animations (`animation-timeline: scroll()/view()`) are a NEW 2025-26 class the pipeline captures in the WRONG (unrevealed) state.** Pure-CSS, no JS, no IntersectionObserver (Chrome 115+, now WebKit) ([Chrome](https://developer.chrome.com/docs/css-ui/scroll-driven-animations); [WebKit](https://webkit.org/blog/17101/a-guide-to-scroll-driven-animations-with-just-css/)). Traps: (i) the 100% keyframe is hit when the element LEAVES the viewport, so authors put "revealed" at ~20-50% — neither scroll-top nor scroll-bottom is the intended look; (ii) the walk runs at `scrollY≈0` → view-timeline elements are at their start (offset/hidden) keyframe. `reducedMotion` does NOT disable these. Fix: neutralize the timeline binding — inject `* { animation-timeline: none !important; animation-range: normal !important; }` (or seek each to mid-point) so it renders at base/declared layout.

**4. Canvas → `toDataURL()` element-level inlining is distinct from region-raster and honors "TRUE element level."** Current capture screenshots a page REGION over a canvas and crops (`cropPng`) — bakes in overlaps and gets dropped by `blankish()` when `preserveDrawingBuffer:false`. Cleaner: in-page per-`<canvas>` `canvas.toDataURL('image/png')` → emit as its OWN image leaf bound to the canvas's exact box ([MDN](https://developer.mozilla.org/en-US/docs/Web/API/HTMLCanvasElement/toDataURL)). Element-true pixels, no neighbor bleed, a real `<img src="data:...">`. WebGL still needs `preserveDrawingBuffer:true`; tainted cross-origin canvases throw → fall back to region-raster.

**5. Multi-STATE (not just multi-pass best-of-text) is how leaders handle the SAME element over time** — live counters, sequential reveals ([abi/screenshot-to-code added video→code](https://github.com/abi/screenshot-to-code); HyperFrames headless-fps composing). Actionable subset for a STATIC V3 page: for a live counter, keep its TERMINAL value ("10,000+"), not the capture-instant value ("0"/mid-tween). Read the element at t0 and after settle, keep the larger; monotone-gate (counters that count DOWN are mis-captured otherwise). Full video-to-motion is OUT of scope — pick the representative terminal frame, motion reproduction is the GSAP-escape-hatch's job (wall C).

**6. `networkidle` as primary goto is an actively-warned-against 2026 SPA anti-pattern** ([Medium](https://medium.com/@gunashekarr11/why-top-automation-teams-avoid-networkidle-and-what-they-use-instead-c0d1e9439dc4); [Apify](https://blog.apify.com/scraping-single-page-applications-with-playwright/)). Analytics/beacons/long-poll keep network non-idle (timeout) or fire idle EARLY before content exists. Refinement: keep `networkidle` only as a capped backstop; add a deterministic content-present `waitForFunction` (text-leaf count / `body.innerText.length > N`) as the PRIMARY gate — self-terminating on content arrival, kills the framer/resend 0.39↔0.80 coverage swing AND speeds simple sites (no fixed 550ms×3). Keep height-stability backstop so it can't capture emptier than today.

### Stream E — AI site-cloners + container/structure inference (competitor techniques + the implementable algorithms)

> Vendor posts are marketing-thin (Locofy 4-model decomposition, Builder 2M-points "studies how components scale", 10Web teardown). The reproducible algorithms live in the academic UI-reverse-engineering literature.

**1. The canonical flat→nested algorithm is recursive XY-cut on projection profiles; the 2025 SOTA fix is exactly Joist's need.** XY-cut recursively splits a region by horizontal/vertical whitespace gaps in the element-bbox projection ([XY-Cut++ arXiv:2504.10258](https://arxiv.org/html/2504.10258v1); [ReverseORC ACM 3445043](https://dl.acm.org/doi/fullHtml/10.1145/3411764.3445043)). XY-Cut++ adds two liftable ideas: (a) **density-driven axis selection** (split horizontally first when dense, vertically otherwise, via ratio `τd`); (b) **pre-masking cross-layout elements** (length > `β·median`, `β=1.3`, AND overlaps ≥2 siblings → mask before cut, re-attach after → cut false splits by 15.9%). Principled version of the manual "overflow/marquee poisons the median-gap" hack. (`τd>0.9`, `β=1.3` are document-tuned — sweep on Joist's corpus, don't copy as constants.)

**2. "Layout dividers" + recursion beats greedy y-band grouping.** Find maximal whitespace gaps spanning full width (→ row boundary) / full height (→ column boundary), recurse into sub-regions. Naturally yields a NESTED tree (row-of-columns-of-rows), more robust to a single tall element than center-overlap banding ([ReverseORC](https://dl.acm.org/doi/fullHtml/10.1145/3411764.3445043)). Over-grouping (a coincidental gutter read as a column boundary) is the dominant failure — commit a grid only at high confidence (trusted `display:grid`+`gridCols`, OR ≥2-3 comparable-width children with full-span dividers); else fall back.

**3. ReverseORC: infer per-element RESIZE behavior (Fill / Hug / Fixed / centered) from ONE desktop pass — the missing Wall-B piece.** ReverseORC reverse-engineers resizable layouts as OR-constraints (per edge: pinned L/R/T/B, centered, or stretches), validated by re-rendering at multiple widths ([ACM 3445043](https://dl.acm.org/doi/fullHtml/10.1145/3411764.3445043)) — the same vocabulary as Figma constraints/resizing (Fixed/Hug/Fill) ([Design+Code](https://designcode.io/figma-handbook-hug-content-resizing/)). Mapping: child.w ≈ parent content-w → **Fill** (`flex:1 1 0` / grid `1fr`); width hugs text → **Hug** (`flex:0 0 auto`); centered with equal L/R margins → **center**+fixed. Joist captures all of this from one desktop pass.

**4. Builder.io's real moat is DATA QUALITY, not a clever grouping algorithm — and they infer all breakpoints from ONE.** They "manually verified that every single bounding box was correct," trained on >2M points, and the model "infer[s] responsive behavior across devices from a single-breakpoint layout" by "studying how components already scale" ([train-ai](https://www.builder.io/blog/train-ai); [Visual Copilot](https://www.builder.io/blog/figma-to-code-visual-copilot)). Implications: (i) Joist owns ground-truth bboxes directly from the DOM (no OCR error) — a structural advantage over screenshot tools; the lever is the grouping/sizing RULES, not box accuracy. (ii) Single-breakpoint→responsive is a solved direction; Joist's Fill/Hug/Fixed classification is the same trick. Every screenshot-to-code tool still fails on the same axis — "aligning elements to fit the original layout" ([aimultiple](https://aimultiple.com/screenshot-to-code)) — which is information Joist already has in the DOM tree.

### Stream B/D — Responsive regrouping primitives + Elementor authoring depth (source-verified)

**1. The container is 100% CSS-VARIABLE-driven — every layout control sets a `--var`, not a direct property.** Source-verified (`flex-container.php`, `grid-container.php`): `direction`→`--flex-direction`, `justify_content`→`--justify-content`, `align_items`→`--align-items`, `gap`→`--gap/--row-gap/--column-gap`, `wrap`→`--flex-wrap`; grid `columns_grid`→`--e-con-grid-template-columns`. **Most load-bearing fact: a Joist-authored container needs NO custom CSS to lay out** — setting the documented settings keys makes Elementor's `post-{id}.css` emit the right vars (the static `e-con` base stylesheet consumes them), and it survives kses untouched (settings JSON, never inline `<style>`).

**2. EXACT responsive control keys** (the thing prior research said "needs GitHub"). `add_responsive_control` generates `{control_id}_{device}`, device ∈ `mobile, mobile_extra, tablet, tablet_extra, laptop, widescreen`; **desktop = unsuffixed base** ([controls-stack.php](https://github.com/elementor/elementor/blob/main/includes/base/controls-stack.php); [breakpoints/manager.php](https://github.com/elementor/elementor/blob/main/core/breakpoints/manager.php)). Group-control prefix is MANDATORY: keys are `flex_direction[_tablet/_mobile]`, `flex_justify_content`, `flex_align_items`, `flex_gap={column,row,unit}`, `flex_wrap`, `grid_columns_grid[_tablet/_mobile]`, `grid_gaps`, `grid_auto_flow`; container base (un-prefixed) `container_type`, `content_width`, `width[_mobile]`, `boxed_width[_mobile]`, `min_height`. Emitting the unprefixed name (`direction`/`columns_grid`) silently no-ops.

**3. Grid columns have a `custom` unit injecting RAW track defs → the RAM responsive-regrouping unlock (Wall B).** `columns_grid` declares `size_units=>['fr','custom']` with `unit_selectors_dictionary['custom']=>'--e-con-grid-template-columns: {{SIZE}}'` ([grid-container.php](https://github.com/elementor/elementor/blob/main/includes/controls/groups/grid-container.php)). Setting `grid_columns_grid={unit:'custom', size:'repeat(auto-fit, minmax(min(<measured>px,100%),1fr))'}` emits a true 3→2→1 reflow with ZERO media queries, ZERO custom CSS, fully native + editable ([web.dev RAM](https://web.dev/patterns/layout/repeat-auto-minmax); [CSS-Tricks auto-fill vs auto-fit](https://css-tricks.com/auto-sizing-columns-css-grid-auto-fill-vs-auto-fit/)). Use `auto-fit` (collapses empty tracks so cards stretch), NOT `auto-fill`. Track-min must be MEASURED `median(cardWidth)` wrapped in `min(<that>,100%)` — a hardcoded constant wraps too early/late and can overflow narrow phones. `mobile_default=>1fr` means Elementor collapses grids to 1 col on mobile out of the box.

**4. The Elementor flex-child width wall is specifically a FIXED-WIDTH wall.** Practitioner-confirmed: fixed `width:33.33%` on flex items "conflicts with flex-grow"; `flex-basis`+`flex-grow`/`flex-shrink` is the reliable reflow path ([Elementor flexbox size behavior](https://elementor.com/help/set-flexbox-container-size-behavior/)). So `flex_wrap:wrap` + per-child `flex:1 1 <basis>` is a DIFFERENT, working responsive path that doesn't fight `.e-con` — the fallback for ragged/non-grid rows; native GRID (finding 3) remains strongest because grid children genuinely honor track sizing.

**5. Frontend `flex-direction` has NO prefix-class (editor-only) → stale CSS is catastrophic for containers.** `prefix_class` for direction only fires `is_edit_mode()`; on the frontend direction is driven purely by the `--flex-direction` var from `post-{id}.css`. If the generated CSS is stale/missing, a programmatic container loses direction/justify/align entirely → **CSS regeneration after write is non-negotiable, and worse for containers than legacy widgets** (containers store ~all layout in regenerated vars). Mobile width is explicitly NOT inherited (placeholders to 100%) — emit `width_mobile`/`boxed_width_mobile` whenever desktop≠full.

**6. Native `motion_fx_*` keys reproduce scroll/parallax WITHOUT GSAP (Pro-only, Wall A/C).** Scrolling Effects (Vertical/Horizontal Scroll, Transparency, Blur, Rotate, Scale) + Mouse Effects serialize as `motion_fx_*` settings (`motion_fx_motion_fx_scrolling`, `motion_fx_translateY_effect`, speed/level 0-10) ([motion-effects](https://elementor.com/help/motion-effects/)). A detected parallax/fade-on-scroll can be approximated NATIVELY and round-trip-editable. Pro-gated — detect Pro first.

**7. Pro Custom Code is the correct GSAP escape hatch — NOT the HTML widget.** Custom Code stores snippets as a CPT with Location + Display Conditions, registrable via `elementor/display_conditions/register` ([custom-code-pro](https://elementor.com/help/custom-code-pro/)); content is NOT run through `wp_kses_post` the way widget HTML fields are. HTML-widget `<script>` is role-gated (`unfiltered_html` only, [#27939](https://github.com/elementor/elementor/issues/27939)) AND has an open render bug ([#20962](https://github.com/elementor/elementor/issues/20962)) — supervised-only for autonomous mode.

### Stream A — Visual/structural fidelity measurement, deeper (2025-26 benchmarks that change the grader's SHAPE + the metric→fix bridge)

> Prior research already banked Design2Code `visual_score.py` internals (Jonker-Volgenant, CIEDE2000 anchors, symmetric area denominator, the <0.5 text-match filter), UI2Code^N RVPO, and the SSIM-saturation root cause. NEW below.

**A1. "VLM judges can RANK but cannot SCORE" — the single most load-bearing new result for the grader objective.** A 2026 conformal-prediction study (3 judges × 14 visual task categories) finds SOTA VLM judges hit only **32–34% exact agreement** on 5-point scales (24–30% off by ≥2), Pearson ρ just **0.30–0.46** for absolute scores — but **ranking is reliable** ("ranking depends only on monotonic consistency, precise scoring requires magnitude calibration"). **Vision-heavy tasks** (fine-grained visual parsing = UI-fidelity) have the WIDEST prediction intervals (3.08–3.50 ≈ 77–88% of the score range); explicit recommendation is *"relative ranking or pairwise comparison is more appropriate"* there ([arXiv 2604.25235](https://arxiv.org/html/2604.25235v1)). This INDEPENDENTLY confirms UI2Code^N's RVPO + ReLook, and tells Joist its **router is the right shape, its absolute-composite threshold is not.** Corollary: the deterministic per-element metric (CIEDE2000/IoU, no VLM) is MORE trustworthy than any VLM-as-judge for the fine-grained channels — Joist's no-OCR typed-tree advantage is doubly validated.

**A2. Vision2Web (2026) quantifies the responsive wall and gives a quantized block-rubric.** Its Visual Score is **block-level**, each functional block scored on a 5-anchor rubric {1.0 / 0.75 / 0.5 / 0.25 / 0.0}, averaged over blocks — a quantized version of Joist's continuous per-element blend, robust (0.66 avg / 0.80 median Spearman to humans). ### Stream B — Dynamic / animated / client-render CAPTURE, deeper (the THREE clocks expand; freeze-and-stabilize)

> Verified against live code: the goto is STILL `waitUntil:'networkidle'` (L60); canvas raster goes through `page.screenshot` region-crop (L692+, NOT `toDataURL`); there is NO `preserveDrawingBuffer` hook and NO `animations:'disabled'` on the raster pass yet. Several "already researched" items are researched-but-NOT-SHIPPED — flagged as quick-bank, not re-proposed. `getAnimations().finish()` + `content-visibility:visible` + step-scroll + `reducedMotion:'reduce'` ARE shipped.

**B1. `timesnap`/`timeweb` virtual-clock is a strictly stronger animation-freezer than `getAnimations().finish()`.** `timeweb` overrides native time sources (`Date.now`, `performance.now`, `requestAnimationFrame`, `setTimeout`, `setInterval`) with a VIRTUAL timeline you advance manually, so JS-driven animation decouples from wall-clock and lands deterministically at any chosen virtual time ([timesnap](https://github.com/tungs/timesnap); [timeweb](https://github.com/tungs/timeweb)). This matters because `getAnimations().finish()` (already shipped) only covers **WAAPI + CSS transitions/animations** — it is a documented no-op for `requestAnimationFrame` loops, Lottie, WebGL shader uniforms, GSAP-without-WAAPI, and `<canvas>` 2D draw loops — exactly the heroes `blankish()` currently drops. A virtual clock advanced to e.g. `t=2000ms` renders a REPRESENTATIVE steady frame that `finish()` cannot touch. Key limit: timeweb does NOT govern pure-CSS keyframes (those need `finish()`/`animations:'disabled'`) — **complementary, not redundant.**

**B2. Double-`requestAnimationFrame` force-paint recovers WebGL/canvas WITHOUT controlling the source.** Independent, additive fix when you can't win the context-creation race: `await page.evaluate(() => new Promise(r => requestAnimationFrame(() => requestAnimationFrame(r))))` immediately before the region-raster — first rAF waits for next render, second guarantees that frame painted into the buffer ([ScreenshotRun teardown](https://screenshotrun.com/blog/puppeteer-playwright-blank-white-screenshots); [WebGL2 tips](https://webgl2fundamentals.org/webgl/lessons/webgl-tips.html)). Use BOTH the `getContext` `preserveDrawingBuffer:true` hook (wave-1 research, still unshipped) AND double-rAF as the universal pre-shot force-paint. Newly surfaced: **WebKit cannot screenshot WebGL2 canvases at all** ([Playwright #17904](https://github.com/microsoft/playwright/issues/17904), [#586](https://github.com/microsoft/playwright/issues/586)) — keep the raster pass on Chromium for canvas regions.

**B3. CDP `Page.captureSnapshot` (MHTML) freeze-and-reload-JS-disabled locks a dynamic page into a stable resting state.** Capture MHTML (serializes iframes, shadow DOM, external resources, inline computed styles), `setJavaScriptEnabled(false)`, reload from `.mhtml` ([Jan Jones — Freezing Puppeteer DOM](https://janjones.me/posts/freeze-puppeteer/); [CDP Page domain](https://chromedevtools.github.io/devtools-protocol/tot/Page/)). Payoff for Joist: with JS off, **scroll-progress transforms, parallax, and rAF jitter stop mutating geometry**, so the `getBoundingClientRect` walk reads a single consistent resting state — directly attacking scroll-position-dependent geometry and the coverage swing at its root because the frozen DOM is identical on every re-read. A DIFFERENT lever than the ensemble (which re-rolls N times); freeze makes ONE capture reproducible.

**B4. CDP `DOMSnapshot.captureSnapshot` returns DOM + layout + computed style + PAINT ORDER in one structured call.** Unlike the per-node `getBoundingClientRect` walk, `DOMSnapshot` returns flattened nodes with layout rects, text boxes, a de-duplicated computed-style table, AND paint order / stacking ([PyCDP DOMSnapshot](https://py-cdp.readthedocs.io/en/latest/api/dom_snapshot.html); [CDP DOMSnapshot](https://chromedevtools.github.io/devtools-protocol/tot/DOMSnapshot/)). This is the documented path to Wall D's blind spot: **abs-positioned / z-index overlay layers the recursive walk SKIPS** — paint order tells you which layer is actually visible at each pixel, so a hero's foreground card over a background canvas is captured as two correctly-ordered leaves instead of one.

**B5. Programmatic IntersectionObserver defeat beats step-scroll for reveal content.** Monkey-patch `IntersectionObserver` in an init script so every `.observe(el)` **synchronously fires the callback with `isIntersecting:true`** and `disconnect()` is a no-op — every lazy/reveal element loads and un-hides at once, no scrolling ([MDN IntersectionObserver](https://developer.mozilla.org/en-US/docs/Web/API/Intersection_Observer_API)). Faster and more complete than scroll-stepping (which misses upward-scroll reveals + custom root margins); pairs with the shipped `content-visibility:visible` override for a single content-complete pass. Keep as an ensemble VARIANT, not a hard default (some sites key real layout off IO; risk of over-revealing mega-menus / modal pre-renders).

**B6. GSAP-specific end-state forcing for scroll-pinned heroes.** No single "finish all" for GSAP, but: `ScrollTrigger.getAll().forEach(st => st.animation && st.animation.progress(1)); ScrollTrigger.refresh();` lands every scroll-driven reveal at its end state without scrolling ([GSAP forum](https://gsap.com/community/forums/topic/41181-scrolltrigger-when-animation-is-triggered-it-needs-to-finish-playing-regardless-of-scroll/); [ScrollTrigger docs](https://gsap.com/docs/v3/Plugins/ScrollTrigger/)). Detect `window.ScrollTrigger || window.gsap`, apply opportunistically — recovers reveal copy `finish()` misses because ScrollTrigger tweens are paused at progress 0 (not "running"), so WAAPI `finish()` never sees them. Feature-detect-guarded → inert when GSAP absent.

**B7. `innerText`-length content-present gate is the field-consensus readiness signal.** Apify/Checkly/ScrapeOps 2025/26 converge: `networkidle` is DISCOURAGED ("can lie if the app keeps background requests open"); the recommended gate is a content assertion via `waitForFunction` — `document.body.innerText.length > N` or a stable container selector ([Apify Playwright 2025](https://blog.apify.com/playwright-web-scraping/); [Checkly waits](https://www.checklyhq.com/docs/learn/playwright/waits-and-timeouts/)). Clean impl: derive N ADAPTIVELY per-site (wait until innerText length is STABLE across two ~400ms samples AND non-trivial), gating capture on real copy rather than the lying network signal at L60/L125. Floor must be adaptive (fixed N under-waits sparse heroes, over-waits text-light visual sites); keep the height-stability backstop.

### Stream E — How AI cloners RENDER recovered dynamic content WELL + EDITABLY (and the implementable structure-inference algorithms)

**E1. Competitors do NOT reproduce live motion — they capture a representative static frame + RE-AUTHOR motion from a fixed vocabulary.** screenshot-to-code literally samples ~20 frames from a ≤30s recording and lets the model pick a steady layout ([abi screen-recording wiki](https://github.com/abi/screenshot-to-code/wiki/Screen-Recording-to-Code)); v0 emits Framer-Motion presets, Framer cascades canned reveal presets ([v0/Lovable/Framer comparison](https://addyo.substack.com/p/ai-driven-prototyping-v0-bolt-and)). **Transferable principle for Joist:** capture the END-STATE faithfully (B1/B2/B6 land the representative frame), render recovered copy as **native text/image widgets at their settled geometry** (never a chunk-screenshot of the revealed region — honors element-level user principle), and reproduce MOTION separately from a small Elementor-Pro Motion-Effects / scoped-custom-CSS preset set (Wall C) rather than transcribing it. Recovered content earns coverage credit ONLY if it lands as real leaves → B1–B6 must feed the per-element scorer, not raster.

**E2. Locofy's production pipeline is 4 deterministic specialist models, NOT one LLM — and its #1 stage is exactly Joist's missing pass.** Locofy LDM ([arXiv 2507.16208](https://arxiv.org/abs/2507.16208), full PDF) = (1) **Design Optimiser** "groups layers… enables higher-level properties such as Auto Layout" (XGBoost on expert poorly-structured→optimized pairs); (2) **Tagging & Feature Detection** YOLO backbone ("Jasmine," 1M+ UI nodes) detects atoms AND "reorganizes and groups components based on visual hierarchy and spatial relationships"; (3) **Auto Components** dedup; (4) **Responsiveness**. The lesson: **flat-geometry→nested-container is a dedicated, separable model stage BEFORE codegen** — Joist's build-absolute/build-flow conflate detection with emission; they should be two passes.

**E3. The deeper-nesting result is the single most important competitive datapoint for the structure pass.** Locofy Figure-10: general LLMs (Llama-2/3.1, T5, Gemini-1.5-Pro, GPT-4o) vs LDM on nesting depth 2→7. **At depth ≥4 every general LLM returns "No result"** — "structurally invalid hierarchies… token limitations, mis-grouping, failure to maintain layout constraints." The deterministic geometric pipeline held across all depths. **Do NOT ask an LLM to emit the nested tree from a flat box list — it collapses past ~3 levels.** A deterministic geometric grouping pass is the correct architecture (cheaper, repeatable, version-stable).

**E4. Locofy's fidelity metric (PMS) is structurally identical to Joist's grader and gives a calibrated 3% threshold.** **Preview Match Score**: per node, `(x,y,w,h)` vs rendered `(x̂,ŷ,ŵ,ĥ)`; a node MATCHES iff all four have abs-pct-error ≤ **3%**; `PMS = matched/total × 100`; **89.6% of 1,000 real screens score >95% PMS** ([arXiv 2507.16208](https://arxiv.org/abs/2507.16208)). Validates Joist's per-element pos/dim grader as the industry-standard objective and offers a concrete `≤0.03` per-attr match predicate (with near-origin denominator guard `max(x, refWidth·0.01)`). Honest gap: PMS ignores text/coverage — the same blind spot Joist already closed.

**E5. 10Web (closest Elementor-native competitor) confirms Container-first decomposition order.** 10Web AI Builder "first identifies and extracts the layout. Then classifies parts based on the **Container** element… Next, **widgets** recreate looks/functionality" ([10web.io](https://10web.io/wordpress-ai-builder/), [help.10web.io](https://help.10web.io/hc/en-us/articles/4413393256210)). Order = **layout → Containers (editable grouping unit) → widgets**, shipped on Elementor with responsive mode — proof the container-tree approach is viable within Joist's EXACT stack and that editability is delivered precisely by emitting real Elementor Containers, not absolute-pinned leaves.

**E6. Builder.io Visual Copilot = separate trained flat→hierarchy model + Mitosis compiler + LLM polish.** A model "trained with over 2 million data points transforms FLAT design structures into code hierarchies," then Mitosis compiles, then a fine-tuned LLM does styling polish; vertical auto-layout→flex-column, horizontal→flex-row, "even if your designs don't follow auto-layout conventions" ([builder.io](https://www.builder.io/blog/figma-to-code-visual-copilot), [best-figma-to-code](https://www.builder.io/blog/best-figma-to-code-plugin)). Same three-stage shape as Locofy: **dedicated structure inference → deterministic compile → LLM only for surface styling.**

**E7. The portable classical core is VIPS (top-down visual-separator segmentation) + a gestalt bottom-up merge.** VIPS ([MSR TR-2003-79](https://www.microsoft.com/en-us/research/wp-content/uploads/2016/02/tr-2003-79.pdf)) recursively segments: detect visual separators (continuous whitespace gaps spanning the block), weight by gap size + font/color contrast + explicit lines/borders, split on highest-weight separator first, recurse until each block's Degree-of-Coherence exceeds a Permitted-DoC stop. A **full-width horizontal separator ⇒ a stacked column split; a full-height vertical separator ⇒ a row split** — recursing yields the nested Container tree directly, geometry-only, depth-robust (the architecture E3 says you want). Complement with gestalt proximity+similarity bottom-up merge ([Springer Visual Intelligence](https://link.springer.com/article/10.1007/s44267-023-00010-1)) where VIPS bottoms out (a card's icon+title+price with no internal separator but tight proximity + shared left edge). PDoC stop threshold is the whole ballgame — tune on corpus, expose as a graded hyperparameter, never a constant. Pitfall: separator weighting needs background/color signal — capture-layout must record per-box bg/border first.

**E8. DCGen residual segmentation gives a no-ML algorithm for Wall D (CSS-drawn / abs / canvas the walk skips).** DCGen detects explicit separators (low-brightness-variance rows whose mean-brightness delta to neighbors exceeds `diff_thresh=45` over `diff_portion=0.9`) AND implicit separators (sliding `window_size=50` zones, variance `<var_thresh=50`, flanked by high-contrast bands), recursively, alternating H/V, **max depth 2** ([arXiv 2406.16386v3](https://arxiv.org/html/2406.16386v3); FSE'25, +15% visual). For Joist this is the region-finder for content the typed walk misses: compute the **residual mask** = source pixels covered by NO matched DOM node, run DCGen banding ONLY inside residual area, emit each leaf band as one positioned `<img>` (region screenshot / `canvas.toDataURL` where untainted) — element-level, never a whole-row chunk. Over-segments on busy backgrounds (gradients, photographic heroes) — cap depth 2, restrict to residual, corpus-gate; an over-banded hero is worse than one clean copied image.

### Pitfalls (this wave 2)

- **Never let any VLM/CLIP/DINOv2 become the optimization TARGET** — the 2026 conformal study is explicit that absolute VLM scores in this vision-heavy regime are unreliable (32–34% exact agreement, intervals to 88% of range). Keep them as gates/guardrails; optimize only the deterministic typed per-element channels.
- **The fix-map (change 3) can chase noise** — a per-element delta is a valid fix signal only on a trustworthy match. Keep the <0.5 text-Dice filter; emit `elementorKey` repairs for high-confidence pairs only; start emit-only, never auto-apply un-gated (a bad match injects a wrong `text_color` write).
- **Zero-reward gate (change 2) must ABSTAIN, not score 0** — a literal 0 contaminates corpus means and ranking; emit `null`/abstain so a temporarily-empty SPA capture isn't baked as a real low grade. Floor must be adaptive (stability-across-samples + modest floor, height backstop retained), or you trade one timing fragility for another.
- **#19528 is a 3.7+ regression, not documented behavior** — verify it still reproduces on sg-host 4.0.9+Pro before banking change 1; emitting desktop-`custom` is harmless either way. Always express reflow as a `_tablet`/`_mobile` OVERRIDE of a PRESENT desktop base — there is no "mobile base"; an absent desktop value cannot be narrowed.
- **Virtual-clock (queued) can deadlock sites that gate first paint on a real `setTimeout`/`requestIdleCallback`** — timeweb governs only JS-driven change; overriding timers globally risks hangs. Always pair with a hard wall-clock timeout + per-site kill-switch; never the default path. It does NOT govern pure-CSS keyframes — keep `finish()`/`animations:'disabled'` for those (complementary, not redundant).
- **`getAnimations().finish()` ≠ rAF/canvas/GSAP-non-WAAPI freeze** — provably a no-op there (why B1/B6 exist). Conversely virtual-clock ≠ CSS-keyframe freeze. Shipping only one leaves a hole.
- **`preserveDrawingBuffer:true` must win the context-creation race** (init script before `goto`); if missed, only double-rAF helps — and even that fails on WebKit WebGL2. Keep canvas rasters Chromium-only; accept "representative, not live."
- **Tainted-canvas blocks `toDataURL`** — any cross-origin texture without `crossOrigin=anonymous` taints the canvas; `toDataURL`/`getImageData` throw SecurityError. This is WHY the `page.screenshot` region-crop path (tainting-immune) MUST stay the canvas mechanism — do NOT switch the pipeline to `toDataURL`; it fails on exactly the third-party heroes you most want.
- **MHTML reload (queued) changes DOM identity** — event handlers, blob: URLs, lazy `<img>` decoders gone after JS-off; geometry should match but canvas-after-load content may go blank. Validate equivalence; restrict to parallax/scroll-pinned sites.
- **IO-defeat / `content-visibility:visible` flood risk** — forcing every observer to intersect over-reveals hidden/duplicate states (mega-menus, modal pre-renders, A/B-hidden variants), inflating coverage with content a human never sees. Keep it an ensemble VARIANT scored by completeness, not a default; keep the existing dark/translucent-bg color guard so flooded hidden layers don't poison color.
- **DCGen residual banding (queued) over-segments on busy backgrounds** (gradients, photographic heroes) — `var_thresh`/`diff_thresh` tuned for flat UI invent false separators in imagery. Cap recursion depth 2, restrict to residual area, corpus-gate; an over-banded hero is worse than one clean copied image.
- **CIEDE2000 stays text+solid-bg only** — ΔE on a single dominant color is lossy for gradients/photos; route imagery to the DINOv2/effects path, not the color channel (prior caveat holds).
- **Settings-only micro-edits without re-capture (queued revise loop) assume the key actually PAINTS** — theme `a{color:…}` beats `text_color` (build-absolute hazard ~L155, links need inline-stamping). Safe for size/weight/padding; for color on links/buttons it is NOT — those need the inline-style stamp + a verification re-capture.
- **Vision2Web's 10–20% desktop→mobile drop is a field FLOOR, not a Joist bug** — don't over-correct abs responsive toward parity with flow; some responsive loss is irreducible. Port flow's per-breakpoint card-row reflow (the identified lever) rather than chasing 1.0.
- **`clear_cache()` is site-wide AND lazy** — at corpus scale, clearing per-build thrashes every page's CSS; prefer warm-GET / per-post enqueue (change 7), and never assume CSS exists at grade-time just because the write returned 200.
- **Global-Kit writes (queued) are site-global** — mutate shared theme state; on a multi-clone farm they bleed tokens between concurrent clones. Scope per-site/per-run, gate behind sandbox isolation.
- **Locofy PMS 3% is strict on small/near-origin nodes** — `|Δx|/x` blows up near x=0; use `max(|Δx|, ε)/max(x, refWidth·0.01)` or an absolute-pixel fallback, else false misses on left-aligned content.
- **Auto-Components dedup (Locofy stage 3) has no clean V3 round-trip primitive** — collapsing repeated cards into a "component" (global widgets are Pro/fragile); keep dedup an internal grouping hint only, still emit concrete per-instance widgets so each logo/card stays its own editable element (honors "true element level").
- **VIPS over-grouping + PDoC sensitivity** (queued structure pass) — too-low PDoC → deep Container chains users hate; too-high → flat blobs; tune on corpus, never hardcode. Separator weighting needs captured bg/border signal — add those to the box-tree FIRST or row/column calls are wrong.
- **None of the Wall-B capture fixes solve Wall-C responsive or the abs narrow-width reflow** — they recover and STABILIZE content; recovered leaves still inherit the desktop-pixel abs limitation until the flow reflow is ported. Bank these as capture/coverage wins, verify each on the section-raster 1:1 proof page before trusting.

### Stream A — Visual/structural fidelity measurement, deeper (race-weighting, GDA group-counts, the quadrant gate, and the structured fix-signal)

Joist has **already implemented the Design2Code shape end-to-end** (Hungarian `linear_sum_assignment`, `<0.5` text-Dice match gate, Sharma-verified CIEDE2000, symmetric area-coverage `matched/(matched+unmatchedSrc+unmatchedClone)`, separate sub-scores, source-vs-source==1.0 self-test) PLUS extensions Design2Code never had (block-merge, global-uniform-y-scale, struct-invariant container gate, effects sub-score, RLG responsive). So this wave does NOT re-propose any of that — it catalogs only what the 2025-26 successors do that Joist still does NOT.

- **A1. WebRenderBench (Oct 2025, [arXiv 2510.04097](https://arxiv.org/html/2510.04097)) replaces Design2Code's per-block-center metric with three render-grounded axes — `R = 0.6·RDA + 0.2·GDA + 0.2·SDA` (RDA relative layout diff, GDA group-wise element-COUNT diff, SDA style diff). Two mechanisms are ABSENT from Joist.** (a) **Race-weighting (`uniqueness`):** each matched pair is weighted `1/(|raceGroup|·C)` where a race group = elements sharing tag+class (repeated cards/nav-links/list-items). This down-weights 12 identical logos / 20 identical pricing cards so they cannot swamp the one-off hero headline. Joist's `perelement-score.mjs` currently weights every matched pair's sub-score equally (`mean()`) and area-weights only the coverage denominator — so a logo-wall counts 12× toward color/typo. (b) **GDA group-count consistency:** for each alignment group (shared left/top/right/bottom edge or centerline), score `1 if |srcGroup|==|cloneGroup| else 0` — a pure COUNT check that catches "source had a 4-card row, clone rendered 3" even when the 3 present cards match perfectly (exactly the Wall-B abs multi-column-overflow failure). Joist's coverage penalizes the missing card's AREA but never asserts COUNT-per-alignment-group — so this is the metric that will MOVE when the abs card-row reflow lands. (c) **3×3 quadrant HARD-gate** on position: matched pair in a different ninth of the page → position=0 outright (sharper than Joist's smooth `1-max(|dcx|,|dcy|)`, which scores a card that drifted across a third at ~0.7).

- **A2. ArtifactsBench (Jul 2025, [arXiv 2507.04952](https://arxiv.org/html/2507.04952v2)) is the converged answer to Wall A's "measure recovered dynamic content": temporal three-step screenshots (before / mid / after a scripted interaction, deterministic seed) + a fine-grained per-task CHECKLIST scored by an MLLM referee, 94%+ human correlation over 1,825 tasks, never one scalar.** The directly-transferable insight: to make `anim-finish`-recovered reveal content EARN score, do NOT sharpen the desktop-pixel metric — feed a before-scroll / post-scroll PAIR to the vision grader with a checklist item *"does the revealed copy render in roughly the right place + color?"* The checklist credits PRESENCE+PLACEMENT, not pixel-exactness — sidestepping the wave-2 trap that anim-finish is "composite-neutral because rendered desktop-pixel imperfectly + barely earns coverage."

- **A3. UI2Code^N (Nov 2025, [arXiv 2511.08195v3](https://arxiv.org/html/2511.08195v3)) independently lands on near-identical weights (Layout 30 / Color 25 / Typography 20 / Spacing 15 / Element-detail 10) to Joist's per-element blend — validating the weighting — and supplies the formal kill-argument against CLIP:** *"CLIP shows limited sensitivity to structural and layout-level discrepancies. Models with substantially different block and position accuracy often obtain similar CLIP scores."* Do NOT add a CLIP/embedding term under schedule pressure — it saturates on exactly Joist's layout/position errors and will INFLATE the score, regressing honesty (confirms wave-1 + wave-2 caveats, now twice-cited). Test-time scaling: real-site fidelity **66%→74% over N=1→5 refine rounds** — concrete payoff evidence for wiring `refine-loop.mjs` to the grader, and the empirical iteration cap (saturates by N≈5 real / N≈3 synthetic; Vision-Guided by 3 cycles).

- **A4. The fix-signal is STRUCTURED-PER-PROPERTY, not holistic — convergent across four 2026 papers, and the holistic-screenshot path under-delivered.** VF-Coder / "Coding with Eyes" ([arXiv 2604.19750](https://arxiv.org/html/2604.19750)): per-component Assert-Color (RGB `d<80`), Assert-Element (presence), layout-similarity asserts drive the fix; lifted Assert-Color 58%→76%, visual 0.428→0.558 — and explicitly *"bug screenshots are helpful but have limited utility"* (the named property asserts, NOT the raw diff image, carried the gains). Vision-Guided Iterative Refinement ([arXiv 2604.05839](https://arxiv.org/abs/2604.05839)): a VLM critic emits STRUCTURED feedback, +17.8% over 3 cycles, 25% distillable into the generator. ReLook ([arXiv 2510.11498](https://arxiv.org/pdf/2510.11498)): generate→diagnose→refine, MLLM critic that LOCALIZES per-element property deviations, designed so code-quality can't diverge from rendered fidelity (anti-reward-hack). 1D-Bench ([arXiv 2602.18548](https://arxiv.org/pdf/2602.18548)): takes an IMPERFECT exported IR + reference render and refines the IR round-to-round via LOCALIZED edits, not regeneration — EXACTLY Joist's situation (captured box-tree = imperfect IR; source render = reference). The engineering pattern is a computed-style diff between two structurally-aligned trees ([RegreCSS](https://github.com/blech75/RegreCSS), [MDN computedStyleMap](https://developer.mozilla.org/en-US/docs/Web/API/Element/computedStyleMap)) — and Joist already OWNS both trees (re-captures the clone via `capture-layout.mjs`, matches to source via Hungarian), so it can emit a per-matched-pair PROPERTY-DELTA record (`{ΔE2000, fontFamilyMatch, fontSizeΔ, lineHeightΔ, letterSpacingΔ, posΔ, sizeΔ, radiusΔ, shadowΔ}`) for free — the artifact `refine-loop.mjs` needs.

- **A5. Web2Code ([arXiv 2406.20098v2](https://arxiv.org/html/2406.20098v2)) renders predicted HTML back to an image AND runs 5,990 yes/no STRUCTURED PROBES — the transferable bit is a small fixed yes/no checklist battery that maps 1:1 onto Joist's existing four human-defect detectors (real-nav / full-bleed / chunked-media / text-collision), suggesting those should be EXPANDED into a uniform checklist array `[{check, srcAnswer, cloneAnswer, pass}]`, not kept as four ad-hoc multipliers.** Cheap, un-saturated, human-aligned, and makes the structural verdict legible/auditable (honors "judge like a human").

### Stream B — Dynamic / animated / client-render CAPTURE, deeper (GPU/ANGLE, video seek, deterministic readiness, settled-frame parallax)

Past wave-2 (anim-finish end-state, step-scroll reveal, `reducedMotion:'reduce'`, lazy-promote, `content-visibility:visible`, `display:contents` flatten, region cropPng). The OPEN sub-walls: WebGL/canvas captured BLANK; no deterministic content-readiness gate so coverage swings; video heroes dropped to a black frame; parallax frozen at the WRONG representative state.

- **B1. Headless Chromium renders WebGL/3D canvases to a BLANK buffer — almost certainly why "Stripe/Framer is the outlier."** Two compounding causes: (a) `<canvas>` with a WebGL context clears its drawing buffer after compositing unless `preserveDrawingBuffer:true` was set at context creation (cannot set on a third-party site) → `toDataURL()` reads empty ([webglfundamentals](https://webglfundamentals.org/webgl/lessons/webgl-tips.html)); (b) headless Chromium has had broken/absent WebGL since Chrome 109–111, especially on ARM Macs — works headful, NOT headless without GPU flags ([Chromium 40062624](https://issues.chromium.org/issues/40062624)). Joist's launch is bare `chromium.launch({ args:['--disable-blink-features=AutomationControlled'] })` (`capture-layout.mjs:55`) — no GPU/ANGLE flags — so WebGL heroes capture as a flat/black box and the cropPng bakes garbage.

- **B2. The fix is real GL via ANGLE, NOT SwiftShader, plus capture-from-the-live-screenshot.** Launch with `--use-angle=gl --ignore-gpu-blocklist --enable-gpu` (createIT measured 8fps→60fps real rendering vs ANGLE-default; SwiftShader is the silent fallback that yields blank/garbage — [createIT](https://www.createit.com/blog/headless-chrome-testing-webgl-using-playwright/), [Promaton](https://blog.promaton.com/testing-3d-applications-with-playwright-on-gpu-1e9cfc8b54a9)). Playwright's NEW headless (chromium channel ≥ v1.49) has materially better GPU support ([Playwright #11627](https://github.com/microsoft/playwright/issues/11627)). On a canvas you don't own you cannot `toDataURL` (blank) — crop the region from `page.screenshot()` while the RAF loop is live (the canvas is visibly composited even though its backbuffer is gone), which Joist already does — but it only works once F1/F2's flags make GL paint at all. Ship a **blank-raster detector** (region variance ≈0 / single dominant color >95%) as the VERIFICATION that GL actually painted (and to stop the grader REWARDING a black canvas).

- **B3. `requestVideoFrameCallback` + seek gives a real poster frame; today video heroes capture black.** `video.currentTime = min(2, duration*0.25)` (or the element's `poster` attr first) → await `requestVideoFrameCallback`/`seeked` → `drawImage`→`toDataURL` ([web.dev rVFC](https://web.dev/articles/requestvideoframecallback-rvfc), [MDN rVFC](https://developer.mozilla.org/en-US/docs/Web/API/HTMLVideoElement/requestVideoFrameCallback)). Seeking is not frame-accurate but fine for a poster; cross-origin video TAINTS the canvas (`toDataURL` throws SecurityError, [MDN CORS image](https://developer.mozilla.org/en-US/docs/Web/HTML/How_to/CORS_enabled_image)) so the page-screenshot region-crop stays the universal fallback (this is WHY cropPng must remain the canvas mechanism — do NOT switch the pipeline to `toDataURL`).

- **B4. Deterministic content-readiness gate — poll `document.body.innerText.length` to a quiet plateau, don't trust networkidle.** Practitioner consensus for SPAs: networkidle is unreliable on chatty apps; gate on a content threshold + a MutationObserver 500ms quiet period, hard-capped ~6s ([apify SPA scraping](https://blog.apify.com/scraping-single-page-applications-with-playwright/), [Chrome MutationObserver](https://developer.chrome.com/blog/detect-dom-changes-with-mutation-observers)). This directly attacks Wall A-(b): a SINGLE deterministic pass beats `capture-ensemble.mjs`'s "run 3×, keep most-text" gamble — turns the 0.39↔0.80 coverage swing into a banked floor, which is the prerequisite that makes the ALREADY-PROVEN anim-finish recovery (+28 leaves/+476 chars) finally BANK into composite.

- **B5. Playwright's `animations:'disabled'` is a cleaner, deterministic end-state lander than the hand-rolled `getAnimations().finish()` loop (fast-forwards finite to completion, cancels infinite to INITIAL state then replays — [Playwright #11912](https://github.com/microsoft/playwright/issues/11912)).** Divergence from Joist's code: the current loop pins INFINITE animations to `delay+activeDuration` (arbitrary mid-state, can catch a marquee mid-slide); Playwright pins them to INITIAL. For looping gradients/marquees, initial is the more representative/neutral frame. ### Stream C/E — Structure inference (Breuel whitespace cuts, XY-Cut++ confirmed constants, Palmer proximity, DBSCAN same-size) + competitor pipelines + the abs-responsive port

- **C1. LayoutCoder's "no-fixed-dims + zero-padding + `flex:[portion]`" is an MLLM PROMPT CONSTRAINT, not a deterministic algorithm — wave-1's "KEY REDIRECT" mis-read it.** Reading [arXiv 2506.10376](https://arxiv.org/html/2506.10376) directly: BFS group-search + 2D block projection + Gestalt division produce only the tree SKELETON; CSS sizing is emitted by an MLLM told "don't impose fixed width/height; margin/padding=0," and `flex:[portion]` comes from `node.get("portion",1)` with NO disclosed formula. **Consequence:** the open "compute flex:[portion] that provably sums-to-100%/no overflow" question was never answered by the source — Joist must supply its OWN deterministic formula, and "strip all fixed dims, zero all padding" is UNSAFE as a blanket rule (it works only for an MLLM that re-derives sizes from the image; Joist has no such step). The correct deterministic analog is **captured-width-ratio → `flex-grow` portions with `flex-basis:0`** (sums-to-whole by construction → no overflow at any width), NOT "delete dimensions and hope." Zero only SYNTHESIZED padding, preserve CAPTURED padding.

- **C2. Breuel's branch-and-bound maximal-empty-rectangle whitespace cover is the rigorous, heuristic-FREE foundation the recursive divider-cut was missing.** "Two Geometric Algorithms for Layout Analysis" (Breuel 2002): finds a globally-optimal cover of background whitespace as maximal empty rectangles via branch-and-bound on a pivot (four sub-problems), *"under 100 lines… globally optimal… requires NO heuristics,"* with an evaluation function that "reliably identifies maximal empty rectangles corresponding to column boundaries." Strictly better-founded than greedy y-band grouping AND than "find the largest gap" — the dominant over-grouping failure becomes a SINGLE tunable threshold on the evaluation-function score rather than a fragile gap heuristic.

- **C3. XY-Cut++ exact params CONFIRMED as published constants (wave-2 filed them as "values to sweep"): pre-mask `T_l = β·median(lengths)`, β=1.3; mask an element iff length > T_l AND it overlaps ≥2 siblings (the "one tall hero poisons the band" violators); axis by density ratio `τ_d = Σ(dense area)/Σ(sparse area)`, τ_d > 0.9 → horizontal split else vertical** ([arXiv 2504.10258](https://arxiv.org/html/2504.10258v1)). Credible defaults now, not guesses — but still site-variable, corpus-gate, never freeze.

- **C4. Palmer/NN/g proximity gives QUANTIFIED grouping thresholds + a design-system gap ladder: <40px grouped 89% of the time, >120px separate 94%; canonical ladder 4/8/16/32/64px = within-control / within-group / between-groups / between-sections / major-divisions** ([NN/g proximity](https://www.nngroup.com/articles/gestalt-proximity/)). Replaces the single hand-tuned "8px alignment tolerance" with a principled two-level classifier (merge <40, hard-split >120, alignment-resolve the 40–120 band) and lets the grader SNAP captured gaps to the nearest tier (de-noises gap medians, tightens fidelity-match) — snap the EMITTED gap only when within a few px of a tier, else keep captured value (avoid visible desktop shift).

- **C5. The fragmented-element problem (Wall D) is a NAMED, hard, actively-researched problem — and the honest takeaway is "do NOT auto-merge."** EGFE (ISSE'24, [arXiv 2309.09867](https://arxiv.org/abs/2309.09867)) + ULMGNN ([arXiv 2412.05555](https://arxiv.org/html/2412.05555v1)) exist because hand-crafted rules to group fragmented vector layers *"perform unsatisfyingly due to visually overlapped and tiny UI elements,"* and even EGFE *"mistakenly group[s] fragmented layers from different merging groups due to overlapping hierarchies."* This VALIDATES Joist's logo-wall-decomposition + "true element level / never chunk a region" principle — SOTA can't reliably merge fragments, so keeping each `<img>`/`<svg>`/`<canvas>` as its own editable element is the SAFE side of an unsolved problem. For Wall D the lever is CAPTURING skipped layers (canvas/abs/CSS-drawn), NOT grouping them.

- **C6. 10Web's "AI Responsive" is the direct competitive proof for Wall B — it RECOMPUTES per-breakpoint CSS-grid track adjustments from the desktop layout, NOT a flow rebuild.** *"Design a desktop layout, click one button, and the AI automatically calculates the exact CSS grid adjustments needed for tablet and mobile… works surprisingly well for complex overlapping elements"* ([max-productive](https://max-productive.ai/ai-tools/10web/)); breakpoints PC >1024 / tablet 1024–767 / mobile <767 ([10Web help](https://help.10web.io/hc/en-us/articles/17464546514578)) — aligned to Elementor's. The competitive lesson: **port flow's per-breakpoint grid-track overrides ONTO the abs tree** (keep desktop pins, add `_tablet`/`_mobile` track recomputes) rather than the crude everything→1col un-pin. **BUT (C5/N9 constraint):** absolute-positioned children inside a CSS grid do NOT respect grid lines for placement (only as a sizing hint) in Chrome/Safari ([MDN](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Grid_layout/Subgrid), [thoughtbot](https://thoughtbot.com/blog/absolute-positioning-with-css-grid)) — so the abs-responsive port must grid-PLACE content widgets (not abs-pin-inside-grid), keeping abs only for true overlay/decoration.

- **C7. Locofy's pipeline stage-ORDER is itself prescriptive and convergent across shipping tools: (1) lift absolute/overlay layers OUT first → (2) convert full-bleed images to backgrounds → (3) group (whitespace-cut) → (4) gaps → (5) alignments** ([Locofy Lightning](https://www.locofy.ai/docs/lightning/design-optimiser/)); FigmaToCode uses an AltNodes IR (manipulable virtual tree) and handles mixed abs+auto-layout via parent-child + z-index detection ([FigmaToCode](https://github.com/bernaferrari/FigmaToCode)). Confirms wave-1's "two passes not one"; the NEW detail is the ordering (lift-abs FIRST also populates the XY-Cut++ pre-mask, preventing the classic "background image grouped as a sibling card"). The universal failure mode is "abstraction level" (flatter-or-overly-granular; element omission/distortion/misarrangement — [aimultiple](https://research.aimultiple.com/screenshot-to-code/)); UI-grouping research shows quality is dominated by a few signals (visual-band priors + color segmentation) and offers a clean primitive: **DBSCAN on normalized bbox (w,h), MinPts=2, eps≈0.0116** clusters same-size elements → high-precision grid-row / auto-component trigger ([arXiv 2403.04984](https://arxiv.org/html/2403.04984v1)). (eps is in NORMALIZED 0–1 coords, not px — normalize by vw/pageH first.)

### Stream A — Visual/structural fidelity MEASUREMENT, deeper (Design2Code source-verified divergences, DiffSpot per-property taxonomy + locality gate, VISTA affine ladder, the learned-metric trap re-confirmed)

Joist already implements the Design2Code shape end-to-end PLUS extensions (block-merge, y-scale, struct-invariant, effects, RLG). This stream catalogs only NEW measurement findings, several pulled FROM SOURCE that correct prior-wave readings.

- **A1. Design2Code's metric is now VERIFIED from `visual_score.py`/`ocr_free_utils.py`, and three implementation details DIVERGE from Joist's port (all auditable, some worth aligning).** ([NoviScl/Design2Code](https://github.com/NoviScl/Design2Code)) (a) **Block detection is OCR-FREE via a color-tagging trick** (recolor each text tag to a unique hex, screenshot twice with a `+50` offset, pixels shifted by exactly 50 ±8 are that block's glyphs — `get_blocks_ocr_free`/`find_different_pixels`); it REQUIRES editing source HTML → impossible for a remote site → Joist's typed-DOM `capture-layout.mjs` is the correct SUBSTITUTE, not a workaround. (b) **The matched color is the average pixel color of the TEXT GLYPHS ONLY** (sampled from the ORIGINAL uncolored screenshot at the block's pixel coords — `average_color`), NOT computed `color`/`background`. Joist feeds computed `color`/`background`, which DIVERGES when text sits on a tint or uses opacity — the glyph-mean is what CIEDE2000 actually scores upstream. (c) **The cost matrix is built ONLY from text-Dice** (`create_cost_matrix` → `-SequenceMatcher.ratio`) with a context bonus (`adjust_cost_for_context`, window=1, bonus=0.1); position/color do NOT enter the assignment, only the per-pair SCORING — Joist's matcher mixing `(1-textDice)+(1-typeMatch)+centerDist` into the COST is a defensible improvement but a documented divergence. (d) **The block-MERGE loop is greedy adjacent-pair merging** (`find_possible_merge`: tentatively merge i with i+1, re-solve, keep if mean-cost improves `>0.05`) — confirms Joist's kept block-merge matches the SOURCE algorithm. (e) **CIEDE2000 normalization anchor is `1 - dE/100`** (dE=100→0); Joist's "full≤2 / zero≥10" perceptual map is FAR stricter — deliberate/justifiable but NOT comparable to published Design2Code color numbers; note when citing. (f) **Final score = `0.2·(size+text+position+color+CLIP)`** (equal-weight 0.2 average) — this CORRECTS the prior-wave "0.2-weights refuted" note, which conflated "components reported separately" with "not equal-weighted." The human-correlated reweight (position-strongest, text-negative) lives in the PAPER's logistic regression ([arXiv 2403.03163](https://arxiv.org/pdf/2403.03163)), NOT the code — so Joist's planned reweight is PAPER-grounded; cite the paper for weights, the code for the algorithm.

- **A2. DiffSpot (2026, [arXiv 2605.29615](https://arxiv.org/html/2605.29615v1)) supplies a citable 13-operator CSS-mutation taxonomy + a grounding-gate primitive reusable BOTH as a delta→fix map AND a deterministic grader self-test.** 13 operators in 4 families: **Typography** {font_weight, font_size, letter_spacing, line_height, text}, **Color** {color, opacity, gradient}, **Layout** {position, spacing, justify}, **Shape** {border, rounded}; difficulty tiers quantified (step-ops: Easy 3–5 Tailwind steps / Medium 2 / Hard 1; continuous: ±0.20 / ±0.12 / ±0.06 em). The **grounding GATE** accepts a mutation only if its pixel diff is CONFINED INSIDE the element's rendered bbox (locality + non-zero-inside predicates). Killer finding: **"neither pixel magnitude nor CLIP distance reliably predicts which property changed"** — the best VLM detects only 40.7% of changes (line_height recall 4%, gradient 26.7%). HARD evidence a pixel/SSIM/CLIP-only grader is STRUCTURALLY BLIND to per-property errors → Joist's typed-tree per-element metric is the right architecture; a VLM-only grader would REGRESS detectability. **The locality gate is the DETERMINISTIC distinctness rail the reverted RLG-struct-mirror lacked** — mutate ONE control on a known-good clone, assert ONLY the matching sub-score moves (scope the assertion to color/typography/shape ops, which DON'T reflow; spacing/position DO reflow neighbors and would false-fail the "only-target-moves" check unless downstream-sibling shift is allowed).

- **A3. VISTA (2026, [arXiv 2605.26144](https://arxiv.org/html/2605.26144v1)) gives a tiered-IoU localization ladder + a per-axis AFFINE alignment that GENERALIZES Joist's single-scalar y-scale fix.** Localization is a 5-tier ladder (IoU≥0.30→1.0, IoU≥0.10→0.6, centerDist≤150px→0.3, ≤600px→0.15, text-fallback→0.1, miss→0), NOT Hungarian. It applies a **per-axis affine `x'=sx·x+tx, y'=sy·y+ty` BEFORE matching** to absorb layout shift — a generalization of Joist's y-only single-scalar `s≈1/hRatio` (Joist does y-only, no translate, no x). Key decoupling result: visual fidelity (CLIP 0.85) and functional correctness (behavior 0.28) are UNCORRELATED across agents → "visual reproduction and functional correctness should be reported separately" — independent confirmation of Joist's multi-term composite (don't collapse to one scalar). Porting x-translate (+ optional x-scale) into the matcher recovers coverage on clones with a global horizontal offset (full-width-canvas edge cases) — BUT a 2-DOF/4-param affine is far more game-prone than 1 scalar: keep `sx`/`sy` independent, cap both to [0.5,2.0], require the SAME anchor-residual validation gate (median residual + inlier-frac) or it over-rescues genuinely-different layouts.

- **A4. Vision2Web (2026, [arXiv 2603.26648](https://arxiv.org/html/2603.26648v1)) + UI2Code^N + DreamSim re-confirm the per-dimension structure AND the learned-metric trap.** Vision2Web's 5-tier component rubric (1.0 exact / 0.75 "<2px misalign" / 0.5 partial / 0.25 recognizable-but-misaligned / 0.0 missing over position/spacing/text/fonts/colors/icons/multimedia) hits **Spearman 0.66 (median 0.80) vs human IAA 0.78** — a VLM rubric judge nearly matches inter-human agreement, the realistic automated ceiling. UI2Code^N ([arXiv 2511.08195](https://arxiv.org/html/2511.08195v1)) shows "RL with CLIP Reward" DEGRADES and switches to a VLM verifier with explicit weights (layout 30 / color 25 / typography 20 / spacing 15 / element-detail 10) — convergent with Joist's blend. **DreamSim** ([arXiv 2306.09344](https://arxiv.org/html/2306.09344v3), [repo `pip install dreamsim`](https://github.com/ssundaram21/dreamsim)) quantifies SSIM-saturation: 2AFC human-preference SSIM 57.3% / LPIPS 70.7% / CLIP 83.1% / DINO 90.1% / **DreamSim 96.2%** (CLIP-ViT-B/32 + OpenCLIP-ViT-B/32 + DINO-ViT-B/16, LoRA-finetuned on NIGHTS), and RETAINS color sensitivity (unlike pure CLIP/DINO) while capturing layout (unlike SSIM). Concrete drop-in over the 0.5·SSIM half — **but learned → adversarially gameable in a generator/grader loop (E-LPIPS warning; DreamSim authors concede backbone bias): SHADOW-only first, human-correlate, flip with the per-element metric as the anti-game anchor.** NEVER gate on a learned visual scalar blind — DiffSpot's CLIP-blindness is the proof a learned scalar lets per-property errors slip.

### Stream A/B — Dynamic / animated / client-render CAPTURE, deeper (commitStyles end-state, CDP DOMSnapshot, double-rAF, in-page toDataURL, multi-frame, three-stage SPA readiness, settled-state parallax)

Past waves: anim-finish end-state, step-scroll, `reducedMotion:'reduce'`, lazy-promote, `display:contents` flatten, region cropPng, GPU/ANGLE flags (proposed), content-readiness gate (proposed). The OPEN sub-walls below are BEYOND those and source-verified against the live `capture-layout.mjs`.

- **B1. `getAnimations().finish()` alone does NOT bake the end-state — you MUST add `commitStyles()`.** VERIFIED against source: the reveal-pass (`capture-layout.mjs` L171–193) sets `a.finish()`/`a.currentTime=end` and the comment CLAIMS content is "committed to opacity:1 / final transform" — but that is only true while the `Animation` object is alive AND has `fill:forwards`. The WAAPI spec is explicit: `commitStyles()` writes the computed current styles into the element's inline `style` so the styling persists after the animation ends; documented pattern is `await anim.finished; anim.commitStyles(); anim.cancel()` ([MDN commitStyles](https://developer.mozilla.org/en-US/docs/Web/API/Animation/commitStyles), [csswg #5394](https://github.com/w3c/csswg-drafts/issues/5394)). Without `commitStyles()`, a NO-FILL reveal (`opacity:0→1` AOS/GSAP entrance — the common case) SNAPS BACK to its underlying CSS (opacity:0) the instant the walk reads `getComputedStyle`/`getBoundingClientRect` after `cancel`, or reports a transient mid-resume value. This is a REAL source of the resend "content recovered but rendered desktop-pixel imperfectly / barely earns coverage" symptom. Commit THEN cancel, per-animation try/catch — forget the trailing `cancel()` on fill-forwards anims and the element is locked out of normal styling.

- **B2. CDP `DOMSnapshot.captureSnapshot` returns BLENDED background color, paint order, and text-box rects NATIVELY — replacing three hand-rolled walks at once.** Field-verified ([CDP DOMSnapshot](https://chromedevtools.github.io/devtools-protocol/tot/DOMSnapshot/)): `includeBlendedBackgroundColors:true` → `blendedBackgroundColors` (the alpha-composited ancestor-walk Joist's `darkAncestorBg()` only APPROXIMATES for dark/solid); `includePaintOrder:true` → `paintOrders` (the z-index/stacking info to know which abs layer is on top — Wall D); `textColorOpacities`; `includeDOMRects:true` → `offsetRects/clientRects/scrollRects`. All computed by the engine's own paint pipeline on the settled page — strictly more accurate than per-element JS `getComputedStyle` walks, and the single biggest "capture the RENDERED truth" lever not yet used. Caveat: the snapshot is a FLAT array keyed through a shared `strings` table → re-association to the JS DOM walk is non-trivial; `includeBlendedBackgroundColors` is EXPERIMENTAL + Chromium-only → land it SHADOW-mode, diff against current `bgOf()` on the corpus before trusting (broadening bg historically FLOODED dark/translucent themes — round-37).

- **B3. Double-rAF force-repaint is the no-control fallback for WebGL/canvas blanks `preserveDrawingBuffer` can't reach.** Even with a `getContext` init-hook forcing `preserveDrawingBuffer:true`, contexts created BEFORE the init script (or re-created by libs) still return blank. The robust external-site fix: `await page.evaluate(()=>new Promise(r=>requestAnimationFrame(()=>requestAnimationFrame(r))))` immediately before the region screenshot — first rAF waits for the next render frame, second ensures that frame actually painted into the buffer ([ScreenshotRun](https://screenshotrun.com/blog/puppeteer-playwright-blank-white-screenshots), [webglfundamentals](https://webglfundamentals.org/webgl/lessons/webgl-tips.html)). WebKit CANNOT capture WebGL canvases at all ([Playwright #17904](https://github.com/microsoft/playwright/issues/17904)) → pin Chromium for raster passes. `preserveDrawingBuffer:true` disables some GPU opts + can shift AA slightly (verify no regression on a working capture) and does NOTHING for `webgpu`.

- **B4. In-page `canvas.toDataURL()` beats post-hoc region cropping for canvas illustrations.** Instead of cropping the canvas region out of the full-page PNG (current S8 path, `capture-layout.mjs` L692–698, which depends on the canvas surviving the composite + is what `blankish()` guards), call `canvas.toDataURL('image/png')` IN THE PAGE with `preserveDrawingBuffer:true` ([webglfundamentals](https://webglfundamentals.org/webgl/lessons/webgl-tips.html), [@zappar/webgl-snapshot](https://www.npmjs.com/package/@zappar/webgl-snapshot)). This reads the canvas's own backing store at NATIVE resolution (no dpr-crop misalignment, no compositing dependency, no overlapping-abs-layer corruption) → a clean PNG uploaded as an `<img>` (element-level, honors "each art element its own image"). CAVEAT: cross-origin-tainted canvases THROW on `toDataURL` → wrap in try/catch, fall back to the rAF region crop; do NOT let the throw abort the walk.

- **B5. Multi-frame / temporal multi-sampling → representative state is the documented answer for INFINITE/looping heroes (which `finish()`/cancel-to-initial capture EMPTY).** abi/screenshot-to-code extracts 20 frames from a screen recording; Replay uses video temporal context to detect animation physics ([screenshot-to-code wiki](https://github.com/abi/screenshot-to-code/wiki/Screen-Recording-to-Code), [Replay](https://www.replay.build/blog/turning-complex-animations-into-production-ready-framer-motion-code-the-definitive-guide)). The cheap deterministic version for Joist is NOT video but **temporal multi-sampling**: for `iterations===Infinity`, sample geometry/raster at currentTime ∈ {0, ¼, ½, ¾}·activeDuration and KEEP the frame whose painted region is MOST-FILLED / LEAST-BLANK (a representative state), replacing the L181–183 "pin to one active-duration end" heuristic. This only PICKS a good still — it does NOT reproduce motion (re-authoring is Wall C, separate).

- **B6. Three-stage SPA readiness (content-present + hydration-mismatch detect) beats both `networkidle` and pure DOM-quiet.** Two NEW signals to add to the converged `domcontentloaded` → DOM-quiet → content-present gate: (a) `waitForFunction(()=>document.body.innerText.length > N)` — the body.innerText>N gate the frontier names as the ROOT fix for the 0.39↔0.80 coverage swing; (b) listen for React/Vue HYDRATION-MISMATCH console warnings — their PRESENCE means the captured DOM is the pre-hydration shell → retry/discard the pass ([alexop.dev](https://alexop.dev/posts/catch-hydration-errors-playwright-tests/), [Firecrawl SPA](https://www.firecrawl.dev/glossary/web-scraping-apis/best-way-to-scrape-single-page-applications-spas) — "you can get a 200, a loaded event, and still extract from an empty shell"). Hydration warnings are framework/build-dependent (stripped in prod) → treat PRESENCE as a strong negative signal, NEVER absence as positive; `innerText>N` is the reliable one. DOM-quiet never settles on perpetual-animation libs → hard timeout cap mandatory.

- **B7. Parallax/scroll-pinned transforms must be NEUTRALIZED, not just scrolled-past.** `getBoundingClientRect` reflects the element's TRANSFORMED box, so an element mid-`translateY` from a scroll handler reports the wrong box even at rest. Playwright "stable" actionability = two consecutive equal animation frames — the same per-element two-frame check can gate the walk. For scroll-linked transforms the only fully-correct capture reads the box WHILE PINNED at the trigger's resting scroll offset → a small per-band re-read (read geometry at the scroll position where each band is at rest) rather than one global read at scrollY≈0. Key reads by element identity + prefer the rest-offset read; fall back to global read when no scroll-transform is detected (else it destabilizes static sites).

- **B8. Competitor convergence on dynamic content (Wall E): nobody round-trips the live animation — capture a representative STATIC state + RE-AUTHOR motion in the target's native motion vocabulary.** Framer treats animations as first-class; v0/Lovable include animations/breakpoints/hover when converting — but the path is design-state→code with motion re-expressed as Framer Motion / CSS, NEVER a faithful replay of the source runtime ([Framer review 2026](https://effloow.com/articles/framer-review-ai-website-builder-guide-2026), [vibecoding 2026](https://www.vibecodingacademy.ai/blog/best-ai-website-builder-2026)). This VALIDATES Joist's strategy: capture the settled representative state faithfully (this stream) and re-author motion separately via Pro Motion-Effects / V4 native interactions / scoped custom-CSS (Wall C). Do NOT try to clone the runtime.

### Stream C/E — Structure inference + the STRUCTURAL FLOOR (the #1 lever): multi-state interaction exploration, structure↔render decoupling, DOM-coordinate grouping, Locofy parent-reassign, archetype priors

This stream is aimed straight at the re-baselined #1 open lever — structural 0.4973, `DYNAMIC-CONTENT-NOT-CAPTURED`.

- **C1. The structural floor is a DETECTION problem competitors solve with a SEPARATE interaction-exploration pass, not a better static parser — WebVIA is the strongest new datapoint.** ([arXiv 2511.06251](https://arxiv.org/html/2511.06251v1), 2026) WebVIA runs an exploration agent in a perception→action→VERIFICATION loop over Playwright: (1) generate candidate actions (clicks/inputs/tab+accordion toggles) from the current DOM, (2) execute, (3) VERIFY each by checking "the action executed AND new interactive elements appeared," building an INTERACTION GRAPH of "rendered screenshots + validated action sequences"; the model is conditioned on MULTIPLE states, not one screenshot. Exploration agent: **89.8% overall / 93.1% completeness** on UIExplore-Bench. ### Pitfalls (this wave 4)

- **CLIP/DreamSim/LPIPS are ALL adversarially gameable in a generator/grader loop** (E-LPIPS NeurIPS 2019; DreamSim authors concede backbone bias) — NEVER gate on a learned visual metric blind; shadow + human-correlate first, flip with the per-element metric as the anti-game anchor. DiffSpot's result (CLIP doesn't predict WHICH property changed; best VLM 40.7%) is the proof a learned scalar lets per-property errors slip.
- **The grader-locality self-test (change 3) MUST be scoped to non-reflowing operators** — DiffSpot only mutates color/typography/shape for clean localization; spacing/position mutations DO reflow neighbors → the "only-target-moves" assertion false-fails unless you allow downstream-sibling shift. This is the same constraint that made RLG-struct-mirror flaky; the deterministic distinctness decision is "scope to color/typo/shape," not "all properties."
- **Glyph-color sampling (change 4) is brittle on anti-aliased/sub-pixel + gradient text** — threshold carefully and fall back to computed `color` when the glyph-pixel count is too low, or it injects noise into a currently-stable color sub-score (re-baseline risk). Shadow-log before folding.
- **`commitStyles()` writes inline `style` which then WINS over everything** — forget the trailing `cancel()` on fill-forwards anims and the element is locked out of normal styling. Commit→cancel, in that order, per-animation try/catch; verify captured text count doesn't DROP on a reveal-heavy AND a static site.
- **`Document::save` vs raw `update_post_meta` is not cosmetic** — a raw write without `wp_slash` corrupts JSON INTERMITTENTLY (only with escaped chars), so it passes smoke tests and fails on real marketing copy with curly quotes/emoji, rendering as the `post_content` fallback = silent floor grade, indistinguishable from a build bug. Prefer `Document::save`; if hand-writing postmeta, encode-THEN-slash is mandatory and asymmetric.
- **`clear_cache()` is site-wide AND lazy AND non-self-healing** — per-build site-wide clears thrash every page at corpus scale; a MISSING `post-{id}.css` is NEVER recreated on render (#7237) so "rendered once unstyled" is a PERMANENT state, not transient. Use per-post `Post_CSS::create($id)->update()` + a warm GET; never assume CSS exists at grade-time because the PUT returned 200.
- **Multi-state click-reveal capture (change 1) can MUTATE state or NAVIGATE AWAY** — a click can open a modal, route to a new page, fire analytics, or surface mega-menus / A/B-hidden variants a human never sees (phantom coverage). Bound depth/breadth HARD, snapshot-and-restore scroll/DOM, require WebVIA's "new visible leaves appeared" verify before keeping, ship report-only first, restrict to in-flow widgets (tabs/accordion/carousel/load-more), dedupe by geometry, keep the dark/translucent-bg color guard. Recovered content also renders desktop-pixel imperfectly today → PAIR with change 2 so recovered subtrees flow into proper containers, not abs-pinned.
- **Structure-decoupling (change 2) risks regressing the kept fidelity recipes** if the structure pass strips fields the emit pass needs — keep the per-leaf fidelity payload (color-stamp, nativeTypo, video, nav, font-register) attached to leaves THROUGH the pass; structure only re-parents/classifies, never drops leaf data. A/B against the struct-invariant grader before promotion.
- **Do NOT ask an LLM to emit the nested tree** — Locofy's depth-ablation: general LLMs return invalid hierarchies at nesting depth ≥4; the deterministic geometric pipeline holds at all depths. Keep `infer-structure.mjs` deterministic; LLM is for surface styling only.
- **Over-grouping remains THE dominant structure-inference failure** (Locofy depth-ablation, VIPS PDoC sensitivity, the flywheel already catching 4 over-eager non-improvements) — DOM-adjacency (queued) is a confidence RAISER, not a license to merge distant bands; commit grids/containers only at high confidence + stay corpus-gated.
- **DOM-coordinate superiority assumes a FAITHFUL captured DOM** — on heavily-shadow-DOM/canvas-rendered sites the DOM is thin → detect "DOM-thin" (few leaves vs visible text) and weight geometry up; otherwise grouping degrades on exactly the dynamic sites the structural floor is about.
- **Genuine `<canvas>`/WebGL/parallax is NOT reproducible editably by ANY shipping tool** (Anima/v0/Lovable all raster-fallback; ~48% layout accuracy on complex interactive) — do NOT let click-reveal success on tabs/accordions tempt a reach for live-canvas reproduction; freeze-to-still + flag for optional Pro Motion remains the honest move (Wall A/D boundary).
- **The delta→control map (queued) must NOT auto-apply position/spacing fixes on the abs builder** — `_position` is not a responsive control and abs offsets are desktop-pixel; a position auto-fix can re-break the mobile reflow wave-4 just landed. Restrict auto-apply to color/typography/border/radius (responsive-neutral); gate position/spacing behind the responsive self-test.
- **MLLM-checklist sub-graders (queued) are NONDETERMINISTIC** — ArtifactsBench needed a DUAL-referee + deterministic seeds + 90.95% agreement to be trustworthy; a single-model checklist drifts run-to-run. Keep report-only until two-referee agreement is measured on the corpus; never let it gate.
- **Don't import Design2Code's flat 0.2 weights as "ground truth"** — the human-correlated weights (position-strongest, text-negative) come from the PAPER's regression, not the code (which is uncalibrated-equal). Cite the paper for the reweight, the code for the algorithm.
- **V4 `styles` ≠ V3 `_tablet`/`_hover` suffixes** — on a V4-authored node, responsive/hover live as VARIANT ENTRIES in the `styles` array, not suffixed keys in `settings`; any code that greps `settings` for `_tablet` keys MISSES V4 responsive styling entirely. Only bites if you read a V4-native node or author V4-atomic directly — but the V4-default lenient-hash regression (re-verify the hash strips `styles:[]`/`interactions:[]`/`editor_settings:[]`) is SILENT: it false-deflates every default-install clone the moment a corpus site is on a fresh V4 stack.
- **Native Entrance Interactions are V4-ONLY** — they live in the atomic `interactions` array and don't exist on V3 widgets; on a V3-on-V4 production tree the safe assumption is they're unavailable for the V3 widgets Joist emits → confirm a V3-widget-inside-V4 can carry an `interactions` entry before banking, else fall back to the Pro Custom-Code hatch (gated on `unfiltered_html`+Pro, silently fails on under-privileged targets).
- **Variables/Global-Class/Component writes are admin-only and SITE-GLOBAL** — PUT `/kit` mutates shared theme state; two concurrent farm runs overwrite each other's tokens → scope per-site/per-run behind sandbox isolation, or keep inlining (the current safe default).
- **8-hex ID standardization must be globally unique across builder re-runs** — coord-seeded IDs collide on re-run → duplicate IDs cross-contaminate `{{WRAPPER}}.elementor-element-<id>` CSS scoping + break sticky/nav JS (#19137/#20302); hash on coords+role+sequence+run-nonce, never reuse.
- **This stream raises CAPTURE faithfulness + per-element color/position sub-scores — the headline composite may move only MODESTLY until the BUILD side renders the recovered content well** (the frontier's stated A-(a) lever); capture fixes without the build/structure half remain the same no-op trap that made standalone anim-finish look dead.

### Stream 1 — Visual/structural MEASUREMENT: Design2Code source re-audit + the color-is-coverage-crushed artifact + CLIP→DINO/DreamSim migration + DiffSpot fix-signal mapping

The canonical `visual_score.py`/`ocr_free_utils.py` were pulled and read LINE-BY-LINE this wave; three of Joist's port divergences are load-bearing and partly EXPLAIN the color floor as a measurement artifact, not only a capture gap.

- **S1-1. Joist MULTIPLIES `areaCoverage` into color/typo/text/position; Design2Code keeps coverage as its OWN equal-weight dimension — this is WHY supabase color reads 0.245 and reactdev 0.17.** Verified `final_score = 0.2·(size + text + position + color + clip)` with size/coverage a SEPARATE term, and color/text/position plain `np.mean` over matched pairs ([NoviScl/Design2Code visual_score.py](https://github.com/NoviScl/Design2Code/blob/main/Design2Code/metrics/visual_score.py)). Joist's headline color = `colorRaw × areaCoverage`, so on a sparse/dynamic section (coverage collapse from Wall A) the color number is COVERAGE-CRUSHED, not raw-color-failure — Joist already emits `raw{}` but the headline conflates "wrong color" with "missing section." The fix is to un-conflate: report coverage in the STRUCTURAL slot, score color/typo/text on the RAW per-pair means. **This will DROP the headline composite on sparse sites — correct behavior per "strictness IS progress," but it MUST land WITH a re-baseline in one supervised step, never piecemeal.**

- **S1-2. Design2Code's matched COLOR is the glyph-pixel MEAN sampled from the original render (`average_color` over text-pixel coords), never computed `color`/`bg` — and its CIEDE2000 anchor is `1 − dE/100`, vastly softer than Joist's `1 − dE/10`.** Joist scores container color from bg-paint and text color from computed `color`; Design2Code only ever scores TEXT-GLYPH color. Joist's `1−dE/10` (zero at dE 10) scores a near-miss brand purple (dE≈6, `#635BFF` vs `#6056F0`) at ~0.4 when a human reads "right purple"; Design2Code's `1−dE/100` scores it 0.94. Neither is obviously right — recommend a PIECEWISE (full ≤ dE2 JND, linear to ~0.3 by dE11 "same family", floor 0 by dE25) validated against a few hand-labeled same/different pairs from the corpus. This is exactly the "grader lied both directions" calibration risk → SUPERVISED.

- **S1-3. Design2Code uses a CONSECUTIVE-context bonus (`adjust_cost_for_context`, window=1, bonus=0.1) + a cost-IMPROVING bidirectional merge (`find_possible_merge`) that is granularity-agnostic — strictly more powerful than Joist's typography-gated block-merge.** The context bonus discounts a match proportional to how many reading-order NEIGHBORS also match (rewards contiguous runs, punishes scattered cross-section matches on repetitive card grids). The merge greedily tries merging each consecutive pair on EACH side and keeps the merge only if it LOWERS matching cost (`difference_of_means`) — where Joist's merge requires identical typography. Both are symmetric (self-test stays 1.0, applied identically both sides). **PERF caveat: `find_possible_merge` is O(n²·solve) per side; cap candidate merges to vertically-adjacent same-column pairs (adjacency already computed) or it dominates grader wall-time on 400-node pages.**

- **S1-4. CLIP is now the named WEAK link in the global-visual term across 2025-26 benchmarks; the migration is to DINOv2 patch features + a VLM-rubric judge, and Joist's SSIM is the same layout-blind analogue.** UI2Code^N: "CLIP primarily captures global visual similarity rather than fine-grained structural correctness" → switched to a VLM-judge 0–100 (threshold 80) + render-feedback ([arXiv 2511.08195](https://arxiv.org/html/2511.08195v3)). IWR-Bench's Low-Level Visual Score = OCR-Levenshtein + **DINO cosine (DINOv2 patch features, NOT CLIP)**, blended 0.5/0.5 with a holistic MLLM score; DINO patches give "stronger structural understanding and spatial localization" than CLIP's global embedding ([IWR-Bench arXiv 2509.24709](https://arxiv.org/html/2509.24709v1); [CLIP vs DINOv2](https://medium.com/aimonks/clip-vs-dinov2-in-image-similarity-6fa5aa7ed8c6)). For Joist, SSIM is the global/layout-blind term DINO would supplement — but a DINO/SSIM term scores a RASTER of the source ~1.0, so it is ONLY honest blended with the editability + structural terms (the same trap `grade-raster.mjs`/grader-v2 fell into). This converges with wave-4's DreamSim shadow proposal; treat DINO and DreamSim as alternative learned high-level terms, SHADOW-first, never gated blind.

- **S1-5. DiffSpot proves a vision/SSIM/VLM grader is STRUCTURALLY BLIND to four properties Joist's typed-DOM metric catches — and that "what changed matters more than where" (pixel/CLIP distance does NOT predict the change, r≈−0.08/+0.06).** Median VLM recall: `line_height` 4%, `gradient` ≤26.7%, `rounded` 13.3% ([DiffSpot arXiv 2605.29615](https://arxiv.org/html/2605.29615v1)). This is the strongest argument FOR Joist's typed per-element DOM metric and AGAINST ever swapping it for a VLM judge "to save effort" — you'd silently lose line-height/radius/gradient/shadow entirely. The convergent production pattern (Cursor Design Mode, Builder.io, ComUICoder, UI2Code^N) is element-WISE feedback: each per-element sub-score delta becomes a TYPED, element-ADDRESSED fix tuple → a localized repair, not a global regenerate. Joist already computes the deltas; it does not yet EMIT them as edits. (Extends wave-4's queued delta→control map with the citation that the DOM metric is the irreplaceable signal.)

### Stream A/B/D — Dynamic / animated / client-render CAPTURE via CDP side-channels (verified NONE shipped) + per-token code color + WebGL/OffscreenCanvas/video freeze + content-readiness

- **AB-1. CDP `Animation` domain is a strictly stronger freezer than page-context `getAnimations()` — it enumerates+seeks compositor-level animations (shadow-DOM, cross-realm, library-wrapped WAAPI, CSS/SMIL) the page-context loop misses, and `setPlaybackRate(0)` halts the document timeline GLOBALLY so newly-spawned anims stay frozen during step-scroll.** `Animation.enable` → collect ids from `animationCreated`/`animationStarted` → `Animation.seekAnimations({animations, currentTime})` to any REPRESENTATIVE pose (not only `.finish()`) + `setPaused`/`setPlaybackRate({playbackRate:0})` ([CDP Animation](https://chromedevtools.github.io/devtools-protocol/tot/Animation/)). This is the productized form of the hand-rolled L185-199 freeze; it's the correct tool for timeline-driven CSS/WAAPI (Framer-Motion/GSAP) where Playwright `clock` actively BREAKS (AB-8).

- **AB-2. CDP `DOMSnapshot.captureSnapshot` returns, in ONE call, the effective BLENDED background color + composited text opacity Chrome computes natively — a direct hit on the per-element color floor (Wall C) that wave-4 only proposed in shadow.** `includeBlendedBackgroundColors:true` → `blendedBackgroundColors[]` ("background colors of all overlapping elements blended"), `includeTextColorOpacities:true` → final composited text opacity, plus whitelisted `computedStyles[]`, `paintOrders[]` (stacking/z-order for the abs builder — Wall D), `offsetRects/clientRects/scrollRects`, and `TextBoxSnapshot` = post-layout rendered text run positions ([CDP DOMSnapshot](https://chromedevtools.github.io/devtools-protocol/tot/DOMSnapshot/)). Replaces the flood-prone `darkAncestorBg()`/`bgOf()` ancestor-walk with the engine's own paint truth. Caveats (same as wave-4 B2): EXPERIMENTAL + Chromium-only; the flat `documents[]/strings`/`layout`/`textBoxes` index arrays are easy to MIS-MAP (shadow DOM flattened, iframes included) and a mis-map injects WRONG colors (strictly worse than the heuristic) → shadow-log + validate against the section-raster 1:1 proof page before flipping; `includeBlendedBackgroundColors` is expensive (call once).

- **AB-3. Syntax-highlighted code blocks are the reactdev color-0.17 KILLER, and the color is literally in the DOM per token — Shiki emits inline `style="color:#…"` per `<span>`, Prism/highlight.js emit per-token classes.** VERIFIED Joist captures `pre`/code as ONE node (`capture-layout.mjs` ~L430-456, single `paint.value` from the `<pre>` root) and `build-absolute.mjs` (~L211) emits ONE `colorCss(n)` → the grader compares one color against a 5-color block → ~0.17. The fix: in the text walk, when inside `pre code`/`[class*="shiki"]`/`[class*="language-"]`/`token`, emit each `<span>` as its own text leaf carrying `getComputedStyle(span).color` (uniform across Shiki/Prism — do NOT build source-specific parsers), keep the `<pre>` as a monospace container, and DON'T surface-raster a `<pre>` that has real token spans ([Shiki](https://github.com/shikijs/shiki); [Prism tokens](https://prismjs.com/tokens.html); [Shiki vs Prism vs hljs 2026](https://www.pkgpulse.com/guides/shiki-vs-prismjs-vs-highlightjs-syntax-highlighting-2026)). Attacks BOTH the color floor AND the structural floor (these code sections are a large share of reactdev/vercel/resend) AND keeps code EDITABLE. **kses/widget-explosion caveat: a 40-line block = hundreds of token spans = hundreds of widgets; MERGE adjacent same-color tokens + cap; inline `style="color:"` survives kses only with `unfiltered_html` (multisite strips it) → preflight + fall back to whole-LINE color if stripping detected.**

- **AB-4. The blank-WebGL problem has TWO composable fixes Joist uses NEITHER of, and `blankish()` deletes exactly what they'd recover.** (a) `addInitScript` hook on `HTMLCanvasElement.prototype.getContext` forcing `preserveDrawingBuffer:true` for webgl/webgl2 BEFORE `goto` (so the screenshot isn't a cleared buffer); (b) the launch flag that actually makes WebGL PAINT headlessly is `--use-angle=gl` (real GPU, ~60fps) vs the swiftshader/ANGLE-software path Joist uses today behind `CAPTURE_GPU=1` (L59, measured ~8fps + frequently blank) — try `--use-angle=gl` first, fall back ([webgl2fundamentals preserveDrawingBuffer](https://webgl2fundamentals.org/webgl/lessons/webgl-tips.html); [createIT headless WebGL via Playwright](https://www.createit.com/blog/headless-chrome-testing-webgl-using-playwright/)). Then loosen `blankish()` so a recovered canvas isn't auto-dropped. ENV-fragile (Docker/CI silently falls back to swiftshader → keep the fallback + a non-blank gate); `preserveDrawingBuffer` does NOTHING for WebGPU ([gpuweb #4356](https://github.com/gpuweb/gpuweb/discussions/4356)) and shifts AA slightly (verify on a working capture).

- **Every new raster path must reuse the per-source filename tag** (`raster-${srcTag}`/`svg-${srcTag}`, L833/839) — the cross-site contamination bug (Stripe wordmark on a pico clone) came from shared `/tmp/svg-N.png` names. Never share temp raster names across sites.
