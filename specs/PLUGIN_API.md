# joist — Plugin API Spec

The WordPress plugin that the AI agent talks to. Every read/write the agent performs against an Elementor site goes through this surface. Designed to be safe under concurrent human editing, schema-aware, and exposable as MCP tools via WordPress's official `mcp-adapter`.

**Status:** v0 draft, 2026-05-10. Core CRUD is grounded in Elementor's documented data model and is unlikely to shift. Extended capabilities (§14) will be expanded after the feature-parity research lands.

---

## 1. Design principles

1. **Elementor's write path is the only write path.** Every mutation funnels through `\Elementor\Plugin::$instance->documents->get($id)->save([...])`. No raw `update_post_meta('_elementor_data', …)`. This guarantees CSS regen, version stamping, hook firing, and slash-handling are identical to a human edit. Grep-banned in CI: `update_post_meta('_elementor_data'` and `get_post_meta(*, '_elementor_data'`.
2. **Optimistic concurrency.** Every read returns a content hash. Every write must echo the hash it was planned against. Mismatch = 409 + current state. No clobbering, ever.
3. **Surgical edits over full replaces.** The agent should patch element subtrees by ID, not re-emit the whole page. Smaller diffs = smaller LLM context = lower cost + lower hallucination surface.
4. **Schema-aware.** The plugin introspects installed widgets via `widgets_manager->get_widget_types()` and exposes their control schemas, so the agent can validate widget `settings` *before* writing — no "invalid widget settings" failure mode.
5. **Round-trip native.** Plugin listens for `elementor/document/after_save` and tags each save with `actor: human | agent` + session ID. The agent sees who touched the page last and re-plans accordingly.
6. **Audit-logged & revertible.** Every write is recorded in a custom revision table with hash-chained tamper detection. Restore is one call. WP's built-in revisions don't cover postmeta — ours do.
7. **Host-resilient.** Detects SiteGround / WP Engine / Kinsta / Cloudways and adapts (auto-flushes SG Optimizer, opens REST through SG Security, etc.).
8. **Dry-run by default for destructive ops.** Patch and replace ops accept `dry_run: true` → returns the diff that would be applied, no write.
9. **Async by default for I/O.** REST controllers never make inline `wp_remote_*` calls. CSS regen, cache flush, webhook emission, frontend verification all defer to `wp_schedule_single_event` or `shutdown` hook. The user sees an immediate response; work completes in the background; a webhook fires on completion or failure. Non-negotiable for shared hosts. See §20 constraint #17.
10. **PolicyGuard refusals are unconditional.** A hardcoded refuse-list of operations the agent role can never perform sits at every controller's entry point, independent of WP capabilities granted. Even an attacker with the agent's App Password cannot bypass it. See §20 constraint #18 and §27.

## 2. Conventions

| | |
|---|---|
| **Base path** | `/wp-json/joist/v1` |
| **Transport** | **HTTPS-only.** Every controller runs `is_ssl()` first; returns 421 `transport.https_required` over plain HTTP. |
| **Auth** | WP Application Passwords (HTTP Basic over HTTPS). Plugin creates a dedicated `joist-agent` user with a custom **reduced-capability role** (NOT Editor by default — see ARCHITECTURE.md §19). Storage of the App Password is delegated to Claude Code's credential manager (macOS Keychain / Linux libsecret / Windows DPAPI); never plaintext in `.mcp.json`. |
| **Required headers (writes from agent role)** | `X-Joist-Session-Id` is required on every write originating from the agent role. Reject 400 if missing — defense against confused-deputy attribution. |
| **Content-Type** | `application/json` for all writes (unless explicitly `multipart/form-data` for file upload). |
| **WP post IDs** | Integers. |
| **Elementor element IDs** | 8-char lowercase hex strings, e.g. `123ab956`. Generated by the plugin on insert if omitted (returned in response as `generated_ids`). |
| **Hashes** | `sha256:<hex>` of canonicalized `_elementor_data` JSON. |
| **Timestamps** | RFC 3339 UTC. |
| **Pagination** | `?per_page=50&page=1`, headers `X-Total`, `X-Total-Pages`. |
| **Idempotency** | Writes accept optional `Idempotency-Key` header (UUID). Plugin caches the response for 24h; replay returns same result without re-executing. |
| **Rate limits** | Per-session token bucket: 30 writes/min, 300 reads/min (configurable). 429 + `Retry-After` header on exhaustion. See §26. |

## 3. Error contract

Every error envelope includes a `recovery_suggestions[]` array of concrete next-call shapes the agent can attempt, ranked by likelihood of success. The MCP server enriches errors before they reach the model, enabling narrate-and-retry rather than dead-end 4xx responses. SKILL.md prompts mandate narrate-and-propose; raw error codes never surface in user-facing output.

```json
{
  "code": "schema.unknown_key",
  "message": "Widget 'heading' has no control named 'justify_content'.",
  "hint": "Did you mean 'flex_justify_content'?",
  "details": {
    "widget_type": "heading",
    "supplied_key": "justify_content",
    "valid_keys": ["title", "header_size", "align", "title_color", "..."]
  },
  "recovery_suggestions": [
    {
      "op": "update_settings",
      "args": {"element_id": "6a637978", "settings": {"flex_justify_content": "center"}},
      "rationale": "Levenshtein-1 + flex_*-aware match"
    },
    {
      "op": "get_widget_schema",
      "args": {"type": "heading"},
      "rationale": "Re-read the schema to find the correct key"
    }
  ]
}
```

For OCC mismatches:
```json
{
  "code": "elementor.hash_mismatch",
  "message": "Page was modified by another writer.",
  "hint": "GET /pages/{id} for current state, then retry with new expected_hash.",
  "details": {
    "current_hash": "sha256:def...",
    "last_modifier": {"type": "human", "user_id": 5, "modified_at": "2026-05-10T22:30:00Z"},
    "diff_summary": ["element 6a637978 settings.title changed", "element abc12345 deleted"]
  },
  "recovery_suggestions": [
    {"op": "get_page", "args": {"id": 123}, "rationale": "Re-read with new hash, then re-plan"}
  ]
}
```

| HTTP | `code` family | Meaning |
|---|---|---|
| 400 | `validation.*` | Bad request shape |
| 401 | `auth.unauthenticated` | Missing/invalid App Password |
| 403 | `auth.capability_missing` | Authenticated but lacks WP cap (e.g. `edit_pages`) |
| 404 | `not_found.*` | Resource gone |
| 409 | `elementor.hash_mismatch` / `lock.held` | Concurrent edit |
| 422 | `widget.invalid_settings` / `schema.*` | Settings don't match widget schema |
| 423 | `lock.held` | Page write-locked by another active session |
| 429 | `rate_limit.*` | Throttled |
| 500 | `internal.*` | Unhandled — full stack in `details` if `WP_DEBUG` |
| 503 | `elementor.missing` / `elementor.incompatible` | Elementor plugin not active / version unsupported |

### 6.5 Container vs section policy

`GET /site` exposes `elementor.layout_mode: "containers_only" | "sections_only" | "mixed"` with confidence score. Autodetect runs nightly: sample 20 pages, classify root elements, set the mode.

**Write-time rule:**
- Parent `elType: section` → child must be `column` (legacy nested) or `widget` (only via column).
- Parent `elType: column` → child must be `widget`, or an `isInner: true` section/container.
- Parent `elType: container` → child must be `container` or `widget`.

Cross-mode inserts at root (container as sibling of section, or vice versa) refused with 422 `layout.cross_mode_refused` unless the patch op includes `force: true` AND the session has an approved Plan Mode plan covering the change. Agent matches parent flavor by default. Helper `POST /pages/legacy-builder/section-with-column` (above) wraps content for legacy sites.

### 6.6 Responsive controls protocol

`GET /widgets/{type}/schema` extends each control with:
```json
{
  "name": "align",
  "type": "CHOOSE",
  "responsive": true,
  "breakpoints": ["desktop", "tablet", "mobile"]
}
```

Suffixed keys: `align_tablet`, `align_mobile`. Custom kit breakpoints (`viewport_md`, `viewport_lg`, plus custom `widescreen` / `laptop` / `tablet_extra` / `mobile_extra` if registered) reflected in the `breakpoints` array.

**Default write policy:** when desktop value differs from the control's default, plugin auto-populates `_tablet` and `_mobile` equal to desktop unless the patch op carries `responsive: "explicit"` (agent claims it has covered breakpoints intentionally).

The success response includes `warnings: [{code: "validation.responsive_incomplete", controls: [...]}]` listing any responsive controls where desktop was set but tablet/mobile were missing (and not auto-populated because `responsive: "explicit"` was set). Non-blocking; informational for the agent.

### 6.7 Custom CSS blocks

The `custom_css` setting on widgets/containers (Pro) is parsed into named blocks:
```css
/* TENET:BEGIN tag=hero_fade_in */
.elementor-element-abc12345 { transition: opacity .3s ease; }
/* TENET:END */

/* TENET:BEGIN tag=user_override */
/* human-written CSS */
/* TENET:END */
```

`GET /pages/{id}` response includes per-element:
```json
{
  "custom_css_blocks": [
    {"tag": "hero_fade_in", "actor": "agent", "session_id": "ses_..."},
    {"tag": "user_override", "actor": "human", "user_id": 5},
    {"tag": "legacy", "actor": "unknown", "preserved": true}
  ]
}
```

`update_settings` op accepts `custom_css_block: {tag, css}` instead of raw `custom_css`. Plugin merges into the existing value, replacing the named block, preserving all others. Unnamed CSS is tagged `legacy` on first parse and never overwritten by the agent. To replace all blocks, use op `replace_element` (intentional and explicit).

### 6.8 Global refs preferred over literals

`GET /widgets/{type}/schema` extends color/typography controls with `supports_globals: true`.

`POST /pages/{id}/patch` with `op: update_settings`:
- If `settings.title_color: "#6334EB"` is supplied AND the value matches a kit global within delta-E < 5, plugin auto-rewrites to `settings.__globals__.title_color = "globals/colors/primary?id=primary"` and returns `transformations: ["color_to_global"]` in the response.
- Agent can suppress with `prefer_literals: true` in the patch op body (rare — e.g. one-off accent colors not in the kit).

New helper:
```json
POST /kit/match-color
{"hex": "#6334EB"}
→ {"global_ref": "globals/colors/primary?id=primary", "delta_e": 2.1, "match_quality": "exact"}
```

Same for typography via `POST /kit/match-typography`.

### 6.9 Inner-element handling

`PatchEngine::insert` auto-infers `isInner: true` based on parent context:
- Parent `elType: column` (legacy hierarchy) + inserted child `elType: section` → child gets `isInner: true`.
- Parent `elType: container` + inserted child `elType: container` with `flex_direction` differing from parent → child gets `isInner: true`.

`SchemaValidator::validateInnerFlag` rejects (`schema.inner_flag_mismatch` 422):
- `elType: section, isInner: false` whose ancestor includes a `column`.
- `elType: container, isInner: false` nested inside another container with a different layout direction.

### 6.10 Deep ID regeneration

`IDGenerator::regenerateTree($subtree, deep: bool = true)`:
- `op: duplicate` → **deep regen** of the entire subtree. Every nested element gets a fresh 8-hex ID. Custom CSS selectors (`.elementor-element-abc12345`), anchor links, and scroll targets must be updated by the agent if they reference inner IDs.
- `op: wrap` → deep regen.
- `op: move` → IDs preserved (move doesn't fork).
- `op: insert` → IDs assigned only where omitted (agent may pre-supply IDs; new IDs are validated for uniqueness against the existing tree).

Response includes `generated_ids: {temp-1: "9988aabb", temp-2: "ccdd44ee", ...}` mapping every placeholder or duplicated ID to its final value. The agent uses this map to rewrite any internal references in copy or custom CSS.

### 6.11 Chained-singleton plan trigger

`SessionTracker` maintains per-session counters:
- `ops_total` since last approved plan
- `ops_destructive` (count of `delete`, `unwrap`, `replace_element` full-tree replaces)
- `ops_per_page[$page_id]`

`PolicyGuard::checkPlanRequired($session_id, $proposed_op)` throws 423 `policy.plan_required` if any:
- Proposed op is `delete` / `unwrap` / full-page-replace (PUT /pages/{id}) AND session has no approved active plan covering this page.
- `ops_total > 5` since last approved plan.
- `ops_per_page[$page_id] > 10` since last approved plan.

Thresholds configurable per-site via `joist_plan_thresholds` option. Session counter resets when a plan covering the page is approved.

### 6.12 Observer / quiet / kill-switch modes

Per-site option `joist_operating_mode`:
- `live` — writes proceed normally (default once user opts in at v1.0+).
- `observer` — all writes return 200 with `dry_run: true` automatically applied; nothing persisted; webhook fires `plan.would_have` event for offline review. **Default for new installs.**
- `quiet` — writes refused for N minutes (timestamp `quiet_until`); returns 423 `operating_mode.quiet`. Use case: agency screen-sharing with a client.
- `kill_switch` — all REST writes refused indefinitely; returns 423 `operating_mode.killed`. Use case: client requested no AI on their site.

Mode togglable in WP admin (one click, visible to all admins and configurable to be visible to clients).

`POST /site/operating-mode`:
```json
{"mode": "quiet", "duration_minutes": 30}
→ {"mode": "quiet", "expires_at": "2026-05-10T23:00:00Z"}
```

Plugin renders the current mode in:
- WP admin top bar badge.
- Elementor editor canvas (banner if `kill_switch` or `quiet`).
- Health check (`GET /health`).
- Webhook `operating_mode.changed` event.

`joist_staging_mandatory` site option — when true, all REST writes refused unless the request `Origin` header matches a configured staging URL pattern. Defaults to false; users opt in per site. Recommended in onboarding docs.

**SSRF defenses (mandatory — see §20 constraint #21):**

- `Max-Redirects: 0` — no redirect following.
- Timeout: 5s connect, 30s total.
- HEAD request first to validate `Content-Type` (must match `image/*` whitelist) and `Content-Length` (must be ≤ `wp_max_upload_size`).
- Post-download MIME-sniff to confirm the content matches the declared type (defense against polyglot files).

## 15. Webhooks (push from plugin → agent)

The plugin can POST events to a configured URL so the agent learns about human edits without polling.

### `POST /webhooks`
```json
{
  "url": "https://agent.example.com/wh",
  "secret": "...",
  "events": [
    "document.saved",
    "document.published",
    "kit.imported",
    "css.regenerated",
    "form.submitted",
    "human.edited",
    "plugin.activated",
    "operating_mode.changed",

  ]
}
```

**SSRF defenses (mandatory — see §20 constraint #21):**
- URL validated at registration AND at each emission (DNS rebinding defense).
- `https:` only.
- Hostname resolves to a public IP (deny RFC1918/loopback/link-local/cloud-metadata).
- Max-Redirects: 0.
- Timeout: 5s.
- Circuit breaker: after 10 consecutive failures, webhook auto-disabled and surfaced as an admin notice. Manual re-enable required.
- Dual-secret rotation supported: `POST /webhooks/{id}/rotate-secret` issues a new secret while the old one remains valid for 24h grace period.

Webhook payload (HMAC-SHA256 signed via `X-Joist-Signature` header, with timestamp in `X-Joist-Timestamp` to defend against replay):
```json
{
  "event": "human.edited",
  "timestamp": "2026-05-10T22:35:00Z",
  "data": {
    "page_id": 123,
    "user_id": 5,
    "before_hash": "sha256:abc...",
    "after_hash": "sha256:xyz...",
    "summary": "Edited 1 widget"
  }
}
```

Receiving end MUST verify: (1) HMAC signature using current OR previous secret (rotation grace), (2) `X-Joist-Timestamp` is within ±5 min of now (replay defense).

## 17. MCP exposure

Every endpoint above is also exposed as an MCP tool via WordPress's `mcp-adapter`, with these naming/shaping conventions:

- REST path `POST /pages/{id}/patch` → MCP tool `elementor.patch_page` with `id` as a required arg.
- Read endpoints become tools named `elementor.get_*` / `elementor.list_*`.
- Write endpoints become `elementor.create_*` / `elementor.update_*` / `elementor.delete_*` / `elementor.patch_*`.
- The plugin registers each as a WP **Ability** (per the Abilities API). `mcp-adapter` discovers and bridges them automatically.

The agent can do **`elementor.list_widgets` → `elementor.get_widget_schema(type)` → construct `settings` → `elementor.patch_page(id, ops, expected_hash)`** as four MCP calls.

## 18. Concurrency & round-trip protocol

The contract that makes human + agent co-editing safe:

1. Agent calls `GET /pages/{id}` → receives `hash_A` and `last_modifier`.
2. Agent plans an edit, calls `POST /pages/{id}/patch` with `expected_hash: hash_A`.
3a. If `current_hash == hash_A` → write succeeds → returns `hash_B`. Agent stores `hash_B` for next op.
3b. If `current_hash != hash_A` → 409. Body includes `current_hash`, `last_modifier` (human or agent + when), and `diff_summary` (which element IDs changed). Agent decides: re-read, re-plan, or abort.
4. For multi-call workflows where the agent doesn't want human interleaving, it acquires a `POST /pages/{id}/lock`. The Elementor editor UI shows a banner; the editor's save endpoint returns 423 to humans until unlock or TTL expiry.
5. Plugin emits `human.edited` webhook on every save where `actor != agent`, so a long-running agent session can react immediately rather than discover stale state on next write.

## 20. Failure-mode design constraints

These rules come from postmortem research of every prior attempt (Wix Harmony, 10Web complaints, Lovable bugs, Elementor MCP issue #32, Builder.ai, abandoned OSS repos). They are NOT optional — every one of them maps to a specific way a competitor's product has broken a real user. Treat as hard product invariants.

1. **Validate every widget write against the live introspected schema.** Never trust an LLM-emitted settings key. Unknown keys → 422 error, not silent passthrough. *(Kills msrbuilds #32: agent wrote `justify_content` instead of `flex_justify_content`, tool returned success, page silently broken on frontend.)*
2. **Read-after-write on every mutation.** Return the post-save element, not `{success: true}`. Confirms the write actually landed. *(Kills msrbuilds #36: tool returned success while save to `_elementor_data` failed silently.)*
3. **Snapshot before every multi-step edit; expose atomic rollback.** Custom revision table; restore is one call.
4. **Surgical diff-based edits only — no full-page regenerate as a tool call.** Agent must target specific element IDs. Refuse "redesign the homepage" as one op. *(Kills Wix Harmony "erased ALL of their work", 10Web "deleted my old page".)*
5. **Auto-flush Elementor CSS + object cache + SG Optimizer after every write, then verify a guest fetch sees the change.** *(Kills msrbuilds #30 §2.3: cache drift means agent thinks page is updated, frontend still serves old version.)*
6. **Token-budgeted reads.** `GET /pages/{id}` defaults to a tree summary; full tree requires explicit flag. Element subtree reads support `depth` + byte cap. *(Kills msrbuilds #30 §3.1: `get-page-structure` returns 100s of KB for large pages, blowing context.)*
7. **Hard pre-flight check.** PHP version, WP version, Elementor version, Pro presence, theme, object cache, CDN, App Passwords enabled. Refuse to operate on incompatible stacks rather than crash mid-edit. *(Kills msrbuilds #18/#19: plugin advertised PHP 7.4 minimum while actually requiring 8.0+.)*
8. **Pin to a tested Elementor version range; gracefully degrade outside it.** Don't expose Atomic-widget tools on v3.x or v3.x widgets on v4.x. *(Kills msrbuilds #28/#29/#38: Elementor 4.0 Atomic broke every tool.)*
9. **Cost meter + per-task spending cap, with cheap-mode fallback.** Don't loop on the same bug; escalate to human after N retries. *(Kills Lovable credit-burn debugging loops; Elementor AI unpredictable credits.)*
10. **Append vs replace must be explicit.** Default for custom CSS, snippets, content is *replace-named-block*, not append. *(Kills msrbuilds #30 §1.2: `add-custom-css` is append-only and accumulates conflicting blocks forever.)*
11. **Tool-count discipline.** Stay under 100 tools (Claude limit) and 128 (Gemini). Prefer one parameterized tool over a dozen specialized ones. *(Kills msrbuilds #27/#37: too many tools → "MCP error -32000: Connection closed".)*
12. **Scope guards.** Agent can only touch elements/sections explicitly in the current task. No "while we're at it" edits. *(Kills Wix/10Web regenerate-and-destroy patterns.)*
13. **Performance budgets enforced at write time.** Reject images > N KB unless compressed; refuse > M animated widgets per page; block known-bloated patterns. *(Kills Spectra/Astra/AI-builder bloat complaints, Lighthouse regressions.)*
14. **First-class export, always available.** WXR + Elementor template JSON + raw rendered HTML + Kit `.zip`. No lock-in path. *(Kills Builder.ai data-loss, Framer no-code-export complaints.)*
15. **Every AI edit tagged in audit log + revision title** ("Edited by Joist Agent — session ses_… — intent: 'Build hero'"). Humans can see and revert exactly what the agent did. *(Kills round-trip mystery edits.)*
16. **Refuse silently-failing operations.** If the underlying API returns ambiguous or partial success (the 10Web / msrbuilds pattern), surface as a hard error, never as success.

The following 14 constraints (#17–#30) were added in the v1 hardening pass (2026-05-10) after independent red-team critiques. Each maps to a real failure mode documented in either a public bug report, wp.org review guidance, security threat model, or agency-user research. See `specs/HARDENING_v1.md` for the full reasoning + critique sources.

17. **Async-by-default for I/O.** No `wp_remote_*` calls or filesystem ops inside REST controller hot path. Defer via `wp_schedule_single_event` or `shutdown` hook. `DocumentWriter::save()` returns optimistically; CSS regen + cache flush + frontend verify run async; webhook fires on completion. *(Kills synchronous-timeout failures on large pages and shared hosts.)*
18. **PolicyGuard refusals independent of capabilities.** Hardcoded deny-list of operations the agent role can never perform regardless of WP capabilities granted: `DELETE /pages/{id}?force=true`, `POST /plugins/install` with `zip_url` outside an admin-initiated session with the `JOIST_ALLOW_ARBITRARY_ZIP` constant, `PUT /kit` with all-zero color palette, `DELETE` on a page where `is_front_page()` AND `status: publish`. *(Kills accidental destructive ops; defense against jailbroken agents.)*
19. **Chained-singleton ops force Plan Mode.** Cumulative ops/session > 5, OR cumulative ops/page > 10, OR any single op of `delete`/`unwrap`/full-replace → 423 `policy.plan_required` until a Plan Mode plan is approved. Thresholds configurable. *(Kills the Plan Mode bypass via N back-to-back single-op patches.)*
20. **HTTPS-only.** Every REST controller checks `is_ssl()`; returns 421 `transport.https_required` over plain HTTP. *(Kills passive credential interception on misconfigured hosts.)*
21. **SSRF defense on every URL input.** `POST /media` (url mode) and `POST /webhooks`: scheme whitelist (`https:` only), public-IP-only resolution check (deny RFC1918 / loopback / link-local / cloud-metadata), re-resolve on connect (DNS rebinding), max-redirect 0, timeout 5s, banned schemes explicitly enumerated. *(Kills SSRF to internal services + cloud-metadata exfil.)*
22. **Custom locks table.** Per-page locks live in `wp_joist_locks` (custom table) with validated `post_id`, explicit TTL column, daily prune cron. NOT in transients. *(Kills wp_options autoload bloat on hosts without persistent object cache.)*
23. **Container-mode matching.** Plugin autodetects layout mode (`containers_only` / `sections_only` / `mixed`). Cross-mode inserts refused without `force: true` and Plan Mode approval. Agent matches parent flavor by default. *(Kills the legacy-site formatting break where agent inserts container as sibling of section at root.)*
24. **Responsive-completeness default.** When desktop control value differs from default, plugin auto-populates `_tablet` and `_mobile` equal to desktop unless `responsive: "explicit"` set in patch op. *(Kills the "looks fine on desktop, broken on mobile" failure mode that would otherwise occur on every responsive control the agent touches.)*
25. **Dynamic tag references must resolve.** Every `__dynamic__` reference validated against the live registered-tags registry; 422 on unregistered. New `GET /dynamic-tags` exposes the registry. *(Kills silent blank-content failures from dangling dynamic tag refs.)*
26. **Global refs preferred over literals.** Agent writing a color/font literal that matches a kit global (delta-E < 5) auto-rewritten to a global ref. Agent may suppress with `prefer_literals: true`. *(Kills brand drift where literal hex values don't respond to kit recolors.)*
27. **Inner-flag inference.** `PatchEngine::insert` auto-sets `isInner: true` based on parent context; SchemaValidator rejects mismatches. *(Kills the "Elementor editor refuses to load" failure mode from wrong isInner.)*
28. **Deep ID regen on duplicate/wrap.** `IDGenerator::regenerateTree(deep: true)` is the default for `duplicate` and `wrap`. Preserve IDs only on explicit `move`. *(Kills nested-ID collisions that break custom CSS selectors, anchor links, scroll targets.)*
29. **Skin-aware schema validation.** `GET /widgets/{type}/schema` returns per-skin control sets for skin-bearing widgets (Loop Grid, Posts, Portfolio, Archive Posts). Settings validated against the selected `_skin`'s schema, not the default. *(Kills the "looks broken because settings were validated against the wrong skin" failure mode.)*
30. **Hash-chained audit log.** Each row's `chain_hash = sha256(prev_row.chain_hash || row_payload_hash)`. Tamper detection even when an attacker with DB access deletes rows. *(Kills silent audit-log erasure by attackers with admin access.)*

These constraints flow into specific spec sections: §6 (page write protocol), §8 (widget validate endpoint), §16 (health check), §18 (concurrency protocol), §23–§31 (v1 hardening additions documented in `HARDENING_v1.md`).

## 21. Output quality & taste constraints

Distilled from May 2026 design-trend research + AI-slop failure patterns. These bias the AI generation pipeline (§19.5, §19.6) and are enforced by the quality gates (§19.15).

### 21.1 The slop tells we reject

Detected by content/code linting on agent output:

| Tell | Why it's slop | Reject by |
|---|---|---|
| `Inter` as the only typeface | Universal AI-builder fallback | Require ≥1 display face from approved foundry list (Pangram, Grilli, Klim) |
| `bg-gradient-to-r from-indigo-500 to-purple-600` or its hex equivalents | The canonical AI-slop gradient | Block `#6366f1`/`#8b5cf6` palette defaults; require monochrome warm-dark + acid accent or earthy + digital pop |
| Uniform 16px radius on every element | Visual flatness | Detect: ≥80% of elements share the same radius → reject |
| Centered hero, single CTA, "Build the future of X" headline | The universal AI hero | Pattern-match generic headline phrases ("Build the future of…", "Scale without limits", "Your all-in-one platform", "Unleash…", "Empower your…") and reject |
| 3-feature-cards-with-Lucide-icons + 3-testimonials + pricing table | Universal AI layout | Flag layouts matching this skeleton; suggest bento variant |
| Stock AI 3D blob or abstract mesh in hero | Filler decoration | Block uploaded AI images whose CLIP score matches "3d abstract gradient blob" reference set; require real photography or product UI |
| Snap transitions, no easing | Missing motion craft | Require easing curve on all transitions; reject `transition: all 0s` |
| Lorem-flavored copy | Concrete-specificity failure | Refuse to write copy without concrete audience + value-prop in the brief |

### 21.2 Positive bias — what we steer toward

**Component sources (in order of preference):**
1. shadcn/ui patterns translated to Elementor widgets/containers
2. Aceternity UI / Magic UI / Cult UI animated landing blocks (motion patterns, not React code)
3. Launch UI / Shadcnblocks for SaaS section composition
4. Tailwind Plus for safe-baseline patterns

**Taste sources:**
- Awwwards SOTD/SOTM last 12 months
- Godly's full catalog
- Land-book featured + SiteInspire minimal/unusual
- Codrops 2026 tutorials for motion patterns
- Mobbin for UX patterns from real shipping apps

**Anti-references — explicitly down-weighted:**
- Envato / ThemeForest Elementor kits (3 years behind)
- Jupiter X bundled templates
- Generic Crocoblock kits
- The default Elementor Library

### 21.3 Typography defaults

Ship the agent with curated pairings (configurable per brand kit):
- Editorial: **Editorial New** (display serif) + **Neue Montreal** (body sans)
- SaaS modern: **Migra** + **PP Mori**
- Brutalist: **Monument Extended** + **Inter** (body only)
- Variable fluid: **GT Ultra Variable**

Avoid as defaults: Inter-only, Poppins, Montserrat, "modern sans" SaaS clichés.

`clamp()` fluid type on all headings. No discrete breakpoints for type scale.

### 21.4 Color defaults

Never default to indigo-500 anything. Three approved palette generators:
- **Warm-dark + acid accent** — charcoal `#0E0E0C` / off-cream / one electric accent (chartreuse / coral / cyan / magenta)
- **Cream + ink** — Pantone 2026 Cloud Dancer `#F3F2EC` background + deep text + restrained accent
- **Earthy + digital pop** — terracotta / sage / ochre / stone + one sharp digital color

Mesh gradients used as ambient *lighting*, never as fill.

### 21.5 Layout defaults

- Asymmetric editorial grids over centered-hero-3-cards
- Bento layouts with varied tile sizes
- Marquee strips, sticky-morph sections, real product UI in hero
- Refactoring UI rules: more whitespace than feels right; hierarchy via weight + color, not size alone; relative spacing scale; one accent used sparingly

### 21.6 Motion budget

- CSS scroll-driven animations (`scroll-timeline`, `view-timeline`) preferred — runs off main thread
- View Transitions API for navigation
- Lenis smooth scroll + GSAP/Framer Motion reserved for hero + section transitions
- **Blocked by default**: full-page WebGL backgrounds, Lottie in hero, parallax on body

### 21.7 The hard rule

> When in doubt, the agent inserts a placeholder ("ADD REAL PRODUCT SCREENSHOT", "REAL FOUNDER PHOTO HERE", "SPECIFIC CUSTOMER QUOTE") rather than generate slop content. Real-content discipline > filled-in-looking page.

## 23. Fleet & bulk operations

Brought forward from v2 SaaS to v1 OSS after agency-user critique — without this primitive, agencies running 10+ sites cannot reasonably adopt.

### CLI

```bash
$ joist connect --config sites.yaml --concurrency 5

# sites.yaml format:
# - url: https://client1.com
#   admin_user: marcus
#   admin_app_password: "xxxx xxxx xxxx xxxx xxxx xxxx"
#   operating_mode: observer        # default for new sites
#   staging_mandatory: true         # optional
#   brand_kit: ./kits/client1.json  # optional pre-configured brand kit
# - url: https://client2.com
#   ...
```

CLI runs each site through the standard onboarding flow (install plugin → create agent user → generate App Password → write claude config) in parallel up to `--concurrency` limit. Per-site outcome reported as a table; failures listed for retry.

### `GET /fleet/status`
Local-fleet endpoint (callable from the CLI host, not site-internal). The CLI maintains a local `~/.joist/fleet.json` registry of configured sites; this endpoint queries each over its REST API and aggregates:

```json

  "sites": [
    {
      "url": "https://client1.com",
      "health": "ok",
      "operating_mode": "observer",
      "open_plans": 0,
      "last_activity": "2026-05-10T20:00:00Z",
      "elementor_version": "3.21.0"
    },
    {"url": "https://client2.com", "health": "warn", "operating_mode": "live", "open_plans": 2, "warnings": ["sg_security.rest_allowed"]}
  ]
}
```

### `POST /fleet/broadcast-brand-kit`
```json
{
  "site_urls": ["https://client1.com", "https://client2.com"],
  "brand_kit_patch": {"voice": {...}, "anti_references": [...]},
  "dry_run": true
}
→ {
  "results": [
    {"url": "https://client1.com", "diff": {...}, "would_apply": true},
    {"url": "https://client2.com", "diff": {...}, "would_apply": true}
  ]
}
```

Applies a partial brand-kit update to N sites in one operation. Default `dry_run: true` so the operator sees the diff before committing. Set `dry_run: false` and re-call to actually write.

### `POST /fleet/broadcast-operating-mode`
Same shape, broadcasts operating mode change (e.g. "quiet for 2 hours during client demo on every site").

Note: v1 fleet operations are CLI-orchestrated and stateless — each call to a site is independent. v2 SaaS adds the persistent control plane with scheduled jobs, alerting, white-label dashboard.

## 24. Operating modes (per-site)

See §6.12 for the per-site mode toggle. This section formalizes the full API:

### `GET /site/operating-mode`
```json
{"mode": "observer", "expires_at": null, "staging_mandatory": false, "set_by": {"user_id": 1, "set_at": "..."}}
```

### `POST /site/operating-mode`
```json
{"mode": "quiet", "duration_minutes": 30, "reason": "Client demo"}
→ {"mode": "quiet", "expires_at": "..."}
```

Capability required: `manage_options` (Administrator). Agent role cannot change its own operating mode.

### Modes
- **`observer`** (default for new installs): every write op returns 200 with the same response shape as if the write had succeeded, plus `dry_run: true` and a webhook emission `plan.would_have`. Nothing persisted. Used for 30-day trial-mode before going live.
- **`live`**: writes proceed normally.
- **`quiet`** (timed): writes refused with 423 `operating_mode.quiet` until `expires_at`. Reads continue.
- **`kill_switch`** (indefinite): writes refused with 423 `operating_mode.killed`. Reads continue. Manual re-enable required.
- **`staging_mandatory`** (per-site flag, can combine with any mode): writes refused unless request `Origin` header matches `joist_staging_url_pattern` regex.

### Client visibility
WP admin renders a top-bar badge showing current mode. By default visible only to admins; site admin can opt to make it visible to all logged-in users (so clients screen-sharing with their agency can see "AI editing is paused").

## 26. Rate limiting

Per-session token bucket enforced at REST controller base. Defaults configurable via `joist_rate_limits` option.

| Class | Rate | Burst |
|---|---|---|
| Writes (POST/PUT/PATCH/DELETE) | 30/min | 10 |
| Reads (GET) | 300/min | 50 |
| AI passthrough (if `/ai/*` ever moves to plugin — current decision: it doesn't) | 10/min | 3 |
| Plugin install | 5/hour | 2 |
| Webhook emission | 100/min | 20 |

429 response includes `Retry-After` header (seconds). MCP server treats 429 as a back-off signal, sleeps, and retries the original request.

Bucket state stored in `wp_joist_rate_limits` table (custom table to avoid wp_options autoload). Pruned hourly.

Agent role + fleet operations honor a *separate* per-app-password budget — `Idempotency-Key` replay does NOT decrement the bucket on cache hit.

## 27. PolicyGuard refusals (unconditional)

Operations the agent role **can never perform** regardless of WP capabilities granted. Enforced at every controller's entry point, before capability check, before rate limit, before any business logic. See §20 constraint #18.

| Operation | Refusal |
|---|---|
| `DELETE /pages/{id}?force=true` from agent role | 403 `policy.force_delete_refused` — agent moves to trash only; admin restores or hard-deletes |
| `POST /plugins/install` with `zip_url` outside an admin-initiated session | 403 `policy.zip_url_disallowed` (also gated by `JOIST_ALLOW_ARBITRARY_ZIP` constant — see §13) |
| `PUT /kit` zeroing the global color palette (all colors empty) | 403 `policy.kit_destructive_refused` — prevents accidental brand wipe |
| `DELETE /pages/{id}` on a page where `is_front_page()` AND `status: publish` from agent role | 403 `policy.front_page_delete_refused` |
| `POST /plugins/{slug}/deactivate` on `elementor` or `elementor-pro` from agent role | 403 `policy.core_plugin_protection` — self-preservation; only admin can deactivate the dependency stack |
| `POST /users/*` (any user CRUD) from agent role | 403 `policy.user_management_refused` — agent has no business touching user records |
| Any write while site is in `kill_switch` mode | 423 `operating_mode.killed` |

The refuse-list is hardcoded in PHP (no DB-driven config) so an attacker who compromises the DB cannot bypass it. Adding entries requires a plugin release.

### `GET /dynamic-tags`

## 29. GDPR / Data Subject Requests

The plugin registers WordPress's standard data-subject-request hooks. Site admins use Tools → Export/Erase Personal Data; our plugin contributes to the export and respects erasure requests.

### Registered exporters
- `joist-audit-log` — exports audit log entries authored by the subject user.
- `joist-revisions` — exports revision snapshots attributed to the subject.
- `joist-sessions` — exports agent sessions linked to the subject (if subject = the admin who approved a plan, sessions they approved show up).

### Registered erasers
- Same set as exporters. Erasure replaces `actor_id` / `approval_user_id` with anonymized placeholder; preserves audit hash chain by re-computing chain_hash forward from the erasure point.

### Retention settings

- Audit log: default 90 days (configurable 30–730).
- Revisions: default keep last 50 per page (configurable 10–500).
- Plans: default keep completed plans 30 days (configurable).
- Bot crawl log: default 30 days.

Daily WP-Cron job prunes expired rows.

When the v2 hosted control plane phones home, EU-flagged sites route to EU-resident infrastructure. v1 plugin doesn't phone home at all, so the question is moot in v1.

## 30. Live preview rendering

Enables side-by-side before/after in the Plan Review UI without writing to the live `_elementor_data`.

### `POST /preview/render`
```json
{
  "page_id": 123,
  "prospective_elements": [/* element tree the agent wants to evaluate */],
  "viewport": "desktop",
  "include_css_diff": true
}
→ {
  "preview_url": "https://example.com/?joist_preview=tok_01HXY...",
  "preview_token": "tok_01HXY...",
```

### Implementation

The Plan Review UI calls `/preview/render` three times (desktop/tablet/mobile) and renders an iframe matrix.

### Decision (v1): preview rendering ON by default. Disable per-site via `joist_preview_enabled` option if DB load is a concern (typically <1MB per preview, auto-cleaned).

## 32. Implementation notes (non-normative)

- **PHP side:** plugin's REST controllers thinly wrap calls to `\Elementor\Plugin::$instance->documents->get($id)->save([...])` for writes, `\Elementor\Plugin::$instance->widgets_manager->get_widget_types()` for schema introspection, `\Elementor\Core\Files\CSS\Post::create($id)->update()` for explicit CSS regen.
- **Revision storage:** custom table `wp_joist_revisions` (id, page_id, hash, snapshot_gzip, actor_type, actor_id, session_id, op_count, created_at). Pruned beyond N revisions per page (configurable).
- **Audit log:** custom table `wp_joist_audit` (id, timestamp, actor, session_id, page_id, op, before_hash, after_hash, payload_gzip, chain_hash — see ARCHITECTURE.md §6).
- **Locks:** custom table `wp_joist_locks` (NOT transients — see §20 constraint #22).
- **Hash canonicalization:** recursive sort of object keys, normalize numeric strings, strip transient `_id` fields the editor adds on render. Defined in `Hasher::compute($elements)`.
- **Hook integration:** `elementor/document/after_save` action tags the save with the active actor (read from request headers `X-Joist-Session-Id` — required for agent role, see §2).
- **Idempotency:** keys cached in transients for 24h. Re-issuing within window returns cached response without re-executing.
- **Async I/O:** CSS regen, cache flush, webhook emission, frontend verification deferred via `wp_schedule_single_event(time() + 1, ...)`. `DocumentWriter::save()` returns optimistically with `pending_verifications: [...]`; verification webhook fires on completion.
