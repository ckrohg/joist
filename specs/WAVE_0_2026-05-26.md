## 0. TL;DR

Three load-bearing surprises between 2026-05-13 and 2026-05-26:

1. **Direct competitor shipped.** Novamira Pro (May 15) — same MCP-adapter foundation, same round-trip claim, same per-site brand memory direction. €49/yr. Our "no direct hit" positioning is dead. Differentiation is now: provable round-trip (16 invariants + acceptance suite as a *public artifact*), multi-page orchestration with a site graph, honest fidelity target, Widget Pack.
2. **WordPress 7.0 "Armstrong" shipped** May 20 with native AI Client, Connectors API, Client-Side Abilities, DataViews/DataForm components, **and** the iframed-editor breaking change.
3. **Elementor 4.0 is default since March 30** with broken atomic-element saves (issues #35888, #35625 still open). Our 3.21 pin is stale. A fresh GrowBig install pulls WP 7.0 + Elementor 4.0.9 today.

Everything else is incremental or confirmatory but still affects scope: Subgrid is Baseline (delete widget), Anchor Positioning is Baseline (add widget), Claude's `memory_20250818` tool replaces our `preference_memory` transport layer, `context: fork` + `CLAUDE_CODE_FORK_SUBAGENT=1` cuts our build-wave compute ~10x, FLUX.2 + per-site LoRA wins image gen, Plan Mode UI bar is DataViews + per-step + blast-radius column.

## 1. The three big surprises (deep)

### 1.1 Novamira Pro — direct competitor

Shipped 2026-05-15. WordPress plugin + MCP server giving Claude Code / Cursor / Windsurf full native access to WordPress, including Elementor atomic widgets, Bricks templates, ACF, JetEngine. Pitch claims: "creates real Elementor containers, widgets, and styles you can edit visually afterwards" — the same round-trip safety claim Joist leads with. Pro tier (€49/yr) adds typed cross-session memory in four categories: User / Feedback / Project / Reference — convergent with our `preference_memory` taxonomy.

**Surfaces they ship:**
- Real-widget output (claim, unverified by us)
- Per-site typed memory (claim, surface unknown)
- Dev/Staging-only restriction with backup discipline (caveat in their docs)
- MCP server endpoint

**Surfaces they don't ship (verified):**
- No multi-page orchestration
- No site-graph
- No public acceptance suite or failure-mode catalogue
- No clone pipeline / DOM→Elementor extraction
- No Widget Pack expanding Elementor's expressive ceiling
- No honest fidelity target / eval loop

**Source:** [novamira.ai](https://novamira.ai/), [GPL Designers review May 16](https://gpldesigners.com/novamira-pro-ai-agent-elementor-bricks-wordpress/), [github.com/use-novamira/novamira](https://github.com/use-novamira/novamira)

**Decision:** Joist's pitch updates from "no direct hit" to "the disciplined option." Specifically, lead with: (a) the 16 (now 20) failure-mode constraints as a public artifact, (b) multi-page orchestration with site graph, (c) honest fidelity target with eval loop, (d) the Widget Pack as a parallel capability layer to Elementor Pro. Stop leading with MCP — that's now commodity infrastructure thanks to WP 7.0.

### 1.2 WordPress 7.0 "Armstrong" — May 20

Core ships:
- Provider-agnostic **AI Client** in core
- **Connectors API** + Connectors hub in dashboard for plugins to register AI providers
- **Client-Side Abilities API** (JS counterpart to server-side Abilities registry)
- Official Anthropic / OpenAI / Google provider plugins
- **DataViews / DataForm** components in `@wordpress/dataviews` — table/grid/list switching, bulk actions, search, filters
- **Iframed editor** — runs in an iframe; legacy admin JS reaching for top-level `document.*` / `window.*` silently fails
- **React 19** with concurrent rendering
- **Classic `add_meta_box()` flips post out of collaboration mode**
- PHP 7.4 floor, 8.3 recommended

**Material to Joist (positive):**
- The MCP-adapter + Abilities API bet from `architecture_decisions.md` is now first-class core machinery, not a bridge
- Register Joist via the Connectors API → users authenticate once, every WP 7.0 AI client sees us
- DataViews + DataForm is the right substrate for Plan Mode UI — *this is the WP-native convention reviewers will expect*

**Material to Joist (negative):**
- Commoditizes "AI connection to WordPress" — our pitch can no longer lead with MCP
- Iframed editor breaks any plugin JS that reaches for `document.*` from outer admin frame — new failure-mode constraint #18
- `add_meta_box()` for plugin UI degrades collaboration — new failure-mode constraint #19

**Source:** [wordpress.org/news/2026/05/armstrong](https://wordpress.org/news/2026/05/armstrong/), [Make WP Core, Feb 24 iframed editor](https://make.wordpress.org/core/2026/02/24/iframed-editor-changes-in-wordpress-7-0/), [WP 7.0 migration playbook](https://dev.to/victorstackai/wordpress-70-iframed-editor-migration-playbook-for-meta-boxes-plugins-and-admin-js-55pm), [DataViews in WP 7.0](https://make.wordpress.org/core/2026/03/04/dataviews-dataform-et-al-in-wordpress-7-0/)

**Decision:** Register Joist as a WP 7.0 Connector for v1.0. Build Plan Mode UI on DataViews + DataForm. Audit our codebase for `document.*`/`window.*` access from outside the editor iframe — `ownerDocument` of a known editor ref, or pure `wp.data` subscriptions only. No `add_meta_box` anywhere in Joist UI.

### 1.3 Elementor 4.0 default + broken atomic saves

Elementor 4.0 became default for new installs 2026-03-30; 4.0.9 is current as of late May. The V4 atomic-element schema diverges from V3 `_elementor_data` in ways that break our existing writes:

- **Issue #35888** (opened May 14, open) — atomic-element saves fail silently with `this.view.container is undefined` and `elementor.documents.getCurrent() is null`
- **Discussion #35627** (open) — "Atomic Editor breaks the site when activated" with no official workaround
- **Issue #35625** (Apr 29, open) — V4 atomic styling embedded in V3 templates throws `Argument #2 ($post) must be of type WP_Post, null given` TypeErrors
- **Issue #24221** (open since 2022) — "Saving doesn't work when too many elements, 500 after 1 minute" — bites on shared hosting

**Our 3.21 pin** also means PHP 8.4 deprecation flood (Elementor 3.32 added the fixes — see Issue #33001).

**Decision:** Pin Elementor to **3.33–3.34.x** for v0.5 smoke test (post-PHP-8.4-deprecation fixes, predates atomic default). Build an Elementor version detector that runs on plugin connect → refuses or adapts based on detected major version. Build an **Elementor 4 atomic-schema adapter** as a v0.85 milestone (between v0.7 and v0.9) so we don't ship v1.0 unable to write to current-default Elementor installs.

**Source:** [Issue #35888](https://github.com/elementor/elementor/issues/35888), [Discussion #35627](https://github.com/orgs/elementor/discussions/35627), [Issue #35625](https://github.com/elementor/elementor/issues/35625), [Issue #33001](https://github.com/elementor/elementor/issues/33001), [Elementor changelog](https://raw.githubusercontent.com/elementor/elementor/main/changelog.txt)

| Widget | Verdict | Note |
|---|---|---|
| 1. Pin-Scroll (shipped) | **Keep, reduce JS scope** | Add `@supports (timeline-trigger: --t)` gate; Chrome 145+ has triggers, JS fallback for Safari/Firefox |

| 6. Reparent | **Ship as planned** | No native DOM-reparenting primitive shipped |
| 7. Display-swap | **Simplify to pure CSS** | Container queries Baseline since 2023; name-only `@container` in Firefox 149 + Safari 26.4; ~30 LOC of generated CSS |

**New widget opportunity:**
- **Anchored Pop** — Anchor Positioning hit Baseline 2026 (Chrome 125+, Safari 26+, Firefox 147+ since Jan 13). Elementor exposes nothing for `position-anchor` / `anchor()` / `@position-try`. **Biggest "browsers have it, Elementor doesn't" gap right now.** Pure-CSS tooltips, dropdowns, callouts, sticky labels tethered to any other widget.

**Other new opportunities:**
- **Transition Group** (Chrome 147 `element.startViewTransition()` + `view-transition-class`) — animate state changes in a section without page nav
- **Auto-Contrast Text** setting (`contrast-color()`, Chrome 147, Baseline April 2026) — guaranteed WCAG contrast against any palette variable
- Skip `border-shape` (Chrome 147 only — wait for cross-browser)

**Structural walls — still standing:**
- Cross-widget Container Queries: `@scope` reached Baseline but doesn't create a container; scroll-state container queries Chrome-only. No movement.
- Cross-widget `:has()`: no CSSWG consensus on shadow-boundary crossing. No movement.

Our existing strategy (single CSS file scoped to body + Elementor's flat DOM) remains correct. **Do not invest in waiting for these.**

**Sources:** [caniuse anchor-positioning](https://caniuse.com/css-anchor-positioning), [caniuse cross-document VT](https://caniuse.com/cross-document-view-transitions), [caniuse subgrid](https://caniuse.com/css-subgrid), [web.dev/blog/interop-2026](https://web.dev/blog/interop-2026), [Chrome scroll-triggered animations](https://developer.chrome.com/blog/scroll-triggered-animations).

### Stream D — Plan Mode UX bar

**Bar to clear (must ship):**
- Editable markdown/checklist plan as primary surface — render via DataViews `list` layout, one row per step
- Per-step row actions: `Edit`, `Skip`, `Run only this`, `Show diff` (DataViews `actions` API with `isEligible` callbacks)
- Per-step diff modal — show Elementor `Document::save()` payload as before/after JSON-tree diff plus thumbnail (generic line-diff is wrong for nested widget JSON)
- Single hard approve at the bottom; nothing writes until that click
- Mode indicator in WP admin bar (Claude Code's plan-mode-leakage bug is the cautionary tale)
- Rollback via Kit `.zip` snapshot before every plan run — surface in UI

**Delight (stretch):**
- **Blast-radius column per step** — "affects 14 pages," "modifies global colors (site-wide)," "deletes Header template (irreversible)." Flagged red when reversibility=Irreversible ∧ visibility=Public. *Nobody else ships this — open space.*
- Iframe live preview with current vs. proposed side-by-side
- Per-step checkbox grid for partial approval + "Run selected" CTA
- Edit step as structured DataForm modal (not free-text)
- Plan-diff between revisions (audit user's own edits)

**Anti-patterns:**
- Single approve/reject with no third option (Claude Code complaint thread)
- Free-text rejection that re-prompts (leaks Claude Code out of plan mode)
- Generic line-diff for nested Elementor JSON
- One approval per micro-mutation (approval fatigue → rubber-stamping)
- Hidden mid-execution mutations
- Chat-only refinement (slower, lossier than direct edit)

## 3. Architecture deltas

### 3.1 `preference_memory` substrate refactor

**Current:** Bespoke storage (per-site tables) + bespoke API + bespoke rendering.

**New:** Anthropic `memory_20250818` tool as transport + Joist-implemented `AbstractMemoryTool` subclass that writes to existing storage. Keep all our value-add (per-site namespacing, dedup, confidence, compact, `renderForPrompt` with 40-rule / 800-token cap) as the *handler implementation*. Anthropic does not provide those semantics.

**Why:** Standard transport means any future Anthropic client (Claude Code, Claude Desktop, third-party SDKs) can read/write our preference store with no Joist-specific glue. Locks our investment in the *semantics*, not the *plumbing*.

**Effort:** Refactor the REST surface of `preference_memory` to look like the `memory_20250818` command set (`view / create / str_replace / insert / delete / rename`). Existing logic remains. ~200 LOC delta.

### 3.2 WP 7.0 Connectors API registration

**New:** Register Joist via WP 7.0 Connectors API. Users authenticate Joist once → Claude Code, Claude Desktop, and any WP 7.0 AI client see it without re-auth.

**Why:** This is the **v1.0 launch differentiator that Novamira does not yet ship**. Only works because WP 7.0 shipped May 20. First-mover advantage on the new core API.

**Effort:** Add a Connector registration class. ~100 LOC + config. Gates on WP 7.0 detection; falls back to existing REST auth on 6.x.

### 3.3 Skill bundle refactor (`/elementor-build`, `/elementor-edit`, `/elementor-audit`)

**New:**
- Use `context: fork` + `agent: Plan` or `agent: Explore` for the research-stream phase of build waves
- Set `CLAUDE_CODE_FORK_SUBAGENT=1` for ~10x cache savings on parallel children
- Add `disable-model-invocation: true` to `/elementor-build` (side-effect-heavy, never auto-fire)
- Use `arguments` frontmatter + `` !`bash` `` dynamic injection to drop argument-parsing prologue (~100 LOC removed)

**Why:** Native Claude Code features make our planned skill scaffolding redundant.

### 3.4 Headless orchestration via Agent SDK Task tools

**New:** Any Joist orchestration code that wraps the SDK uses `TaskCreate/Update/Get/List`, not `TodoWrite` (deprecated in SDK 0.2.82 — May 15).

### 3.5 Long-running build-loop pattern

**New:** Initializer session → checkpoint → resume, using server-side Compaction + the memory tool, rather than client-side history pruning. Matches Anthropic's "Multi-session software development" guidance.

### 3.6 Cost-model documentation for June 15 billing split

**New:** Document that users running Joist via headless `claude -p` will hit the separate Agent SDK credit pool starting June 15. Add `JOIST_USE_API_KEY` escape hatch in plugin settings → heavy agencies bypass subscription pool entirely.
